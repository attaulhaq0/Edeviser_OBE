# Bugfix Requirements Document

## Introduction

Multiple UI consistency defects have been observed across the Edeviser admin dashboard and are suspected to affect the Coordinator, Teacher, Student, and Parent role surfaces as well. The defects share a common theme: rendered pages and components diverge from the established design system rules, producing invisible text, rogue gradients, broken card composition, unprofessional empty states, and black surfaces where the platform standard is white.

The user has provided screenshots demonstrating the following symptoms on Admin:

- Faded "Schedule" panel and "Bonus Categories" grid where only the Bronze/Silver/Gold pills are visible (titles, dates, labels invisible).
- A primary CTA button labeled "Add Slot" rendering a non-brand gradient while the sibling "Add ILO" button renders the correct brand teal→blue gradient.
- Section cards ("Users by Role", "Recent Activity") showing a white strip above their gradient header instead of the gradient being flush to the top edge.
- KPI cards (TOTAL USERS, ACTIVE USERS, PROGRAMS, COURSES, ONBOARDING) displaying a bare "0" with no empty-state framing.
- A "Create Bonus XP Event" dialog rendered with a black surface and white text instead of the platform-standard white surface with dark text.
- Bonus XP and Badge Spotlight pages where section headings are not visible.
- Marketplace table whose column headers only appear as a grey highlight on hover.
- Users table where names, emails, and column headers are invisible and only Role/Status badges are rendered.
- A large solid black area beneath the "Graduate Attributes / Historical Evidence / Competency Frameworks" cards, with a barely-visible onboarding checklist widget in the bottom-left.
- Dark mode rendering incorrectly across the platform: primary CTAs showing black text on the brand gradient, card body text becoming invisible against dark backgrounds, and gradients themselves appearing inconsistent between surfaces. The user has confirmed that many of the contrast problems above were observed while the app was in dark mode and that current dark mode is considered broken. Dark mode must be fully fixed to match light-mode polish — disabling dark mode is not an acceptable resolution.
- The `/login` page exposes only a "Sign in" flow and contains no "Create account" / "Sign up" / "Register" link or button, so new users cannot self-register from the UI.
- Once authenticated, users on every role dashboard (Admin, Coordinator, Teacher, Student, Parent) have no way to sign out: there is no sign-out button, link, or menu item rendered anywhere in the dashboard chrome.
- The top header exposes a language-change control but does not render a profile control (avatar + name + dropdown) next to it, so there is no discoverable UI entry point to view the profile or sign out.
- The dashboard renders a persistent "Quick Start" checklist widget pinned at the bottom of the page (with a "0/4 completed" counter visible in the earlier Graduate Attributes screenshot) instead of launching a first-time guided tour overlay that introduces each role to the key surfaces of its dashboard in a proper tour/walkthrough experience.
- Dark mode must be fixed completely rather than disabled. The earlier "disable dark mode as a fallback" escape hatch is rejected; the only acceptable outcome is a fully polished dark theme with correct token swaps across every surface.
- The "Quick Start" / onboarding checklist widget pinned at the bottom of the dashboard is the wrong pattern and should be replaced with a first-login guided tour per role (Admin, Coordinator, Teacher, Student, Parent), with element-highlight + tooltip spotlight behavior, multi-step multi-page support, per-user persistence, restart from profile/help menu, RTL and dark-mode support, and `prefers-reduced-motion` compliance.
- The "Remind" button in the admin dashboard's "Pending Onboarding" section (shown next to rows such as "Demo Student, student@demo.com, Joined 3 months ago, Overdue") is suspected to be non-functional, a no-op, or missing a backend handler end-to-end (click handler, mutation, Supabase persistence, RLS scope, success and error toasts).
- The top banner on every role layout is missing a Settings entry point at the absolute top-right; the expected right-edge order is language switcher → profile dropdown → Settings icon button (gear icon) flush right, and clicking the gear should open the existing role settings page.
- Student onboarding wizard opens at step 4 (`baseline_select`) instead of step 1 (`welcome`) for a first-time login, because seeded progress state is mid-flow and there is no first-time-login detection.
- The "Quick Start" checklist is rendered as a small bottom-of-page widget rather than as a first-time guided tour that walks the user through the dashboard.
- No dedicated Settings icon in the rightmost position of the top bar across any role layout.
- The "Remind" button on Pending Onboarding has no verified end-to-end loading/success/error UX and no idempotency guard.
- Profile pages have no "Upload profile picture" control, and representative surfaces (top bar, sidebar, tables, feeds) do not consistently render users' uploaded avatars.
- Database constraint errors (observed: `badge_spotlight_schedule` `week_start_is_monday` CHECK violation) surface raw PG messages instead of being prevented at the input layer and mapped to user-friendly field-level guidance.
- Student clicks "Complete & Go to Dashboard" at the end of the onboarding wizard and then stares at an unchanged screen for multiple seconds (observed 2–10s) with no spinner, no skeleton, and no optimistic navigation before the Student dashboard finally appears.
- Only the Student dashboard renders a personalized "Good morning, {name}" hero card; Admin, Coordinator, Teacher, and Parent dashboards jump straight into KPI rows / section cards with no greeting hero parity across roles.
- List and grid views across the platform (Courses, Progress, and other lists) render a completely blank content area when they have zero items, with no empty-state illustration, explanatory copy, or next-action CTA.
- The habit heatmap legend inside the Student "All habits" tracker card is clipped on the left edge — color swatches are partially hidden and the "4+ habits" caption is visually detached from its leading swatches because the legend overflows the card's content padding.
- Hovering across habit heatmap cells causes repeated jitter and tooltip flicker — cells re-mount the tooltip on every `mousemove`, there is no `transition-transform` on the cell hover, and the Radix tooltip lacks smooth open/close animation between adjacent cells.
- Switch/toggle controls bound to backend mutations (Wellness tracking opt-in, Public profile) feel laggy: the thumb slide is delayed until after the network round-trip, the track color swap is abrupt, and there is no optimistic visual feedback while the write is in flight.
- When the user switches the locale to Arabic, many page titles, section headings, button labels, empty-state copy, toast messages, and table headers remain in English across multiple roles; the RTL flip works but translation coverage has large gaps, and formal Qatar MOEHE-aligned terminology for OBE/accreditation concepts is not enforced.

The impact is significant: admin and (likely) other-role users perceive the platform as broken, empty, or unfinished on pages that actually have data, and cannot operate core flows (reading tables, identifying primary actions, completing dialogs, registering, signing out, navigating to their profile, first-time onboarding / tour UX) with confidence. This fix must enforce the existing design system globally across all five roles (Admin, Coordinator, Teacher, Student, Parent) so that no rendered page violates the documented tokens, component patterns, and contrast rules, and must close the authentication-surface gaps (dark mode, self-registration, sign-out, profile placement) that currently block basic user flows. The newly captured items widen the bug condition to include the student onboarding first-time flow, the platform-wide first-time experience (a guided tour across all five roles), the top-bar settings affordance, critical admin-to-student communication (the Remind action), user identity representation (avatars across every surface that renders a user), and form/error hygiene (global database-constraint-error mapping and input-level prevention). The bug condition is further widened by the latest production observations: performance of critical transitions (onboarding → dashboard navigation and Student dashboard cold-load), parity of the greeting hero across every role dashboard, professional empty-state UX on every list/grid/table, micro-interaction polish (habit heatmap hover smoothness, optimistic switch/toggle animations, legend-clipping inside cards), and complete Arabic locale coverage aligned with Qatar Ministry of Education and Higher Education (MOEHE) conventions are now all part of the bug condition C(X) and therefore subject to Fix Checking and Preservation Checking.

## Bug Analysis

### Current Behavior (Defect)

### Current Behavior (Defect)

The following symptoms are observed on the Admin role in production-equivalent builds, and the scope of this bug condition extends to every rendered page across every role until audited and proven clean. Clauses 1.1 – 1.32 enumerate the observed defects; clauses 1.27 – 1.30 were added 2026-05-10 after PR #124 audit feedback to cover the global header layout (full-width, logo-leftmost, primary-nav-below), removal of the duplicate top-bar settings gear, end-to-end role-based notification bell wiring, and RTL correctness of the new header. Clauses 1.31 – 1.32 were added shortly after to cover the duplicate sidebar Profile entry (single-entry-point parity with clause 2.28) and the narrowed root-cause specification of the admin-dashboard black-area regression (the Admin-specific instance of clause 1.9).

1.1 WHEN the admin dashboard renders the "Schedule" panel and the "Bonus Categories" grid THEN the system displays headings, date labels, and body text in colors with insufficient contrast against the white card background, making the content appear blank or ghosted.

1.2 WHEN a primary CTA button renders (for example "Add Slot" on the Game Slots page) THEN the system applies a non-brand gradient that differs from the approved brand teal→blue gradient used by sibling buttons such as "Add ILO".

1.3 WHEN a section Card with a brand gradient header renders (for example "Users by Role" or "Recent Activity" on the admin dashboard) THEN the system shows a white strip above the gradient header instead of the gradient being flush to the top edge of the Card.

1.4 WHEN a KPI card has no data to display (for example TOTAL USERS, ACTIVE USERS, PROGRAMS, COURSES, or ONBOARDING on the admin dashboard) THEN the system displays a bare numeric "0" with no supporting copy, producing a broken/empty visual state.

1.5 WHEN a modal, dialog, sheet, popover, or dropdown menu opens (for example the "Create Bonus XP Event" dialog) THEN the system renders the surface with a black background and white text instead of the platform-standard white surface with dark text.

1.6 WHEN the Bonus XP and Badge Spotlight pages render their section headings THEN the system displays heading text that is not visible against its background (white text on white, or a missing gradient header that would otherwise provide dark contrast).

1.7 WHEN the Marketplace table renders its column header row THEN the system displays header labels that are invisible by default and only become visible as a grey highlight while the user hovers over the header cell.

1.8 WHEN the Users page table renders non-badge cells (user names, emails, and other text columns) and its column header row THEN the system displays text that is invisible against the white row background, and only Role and Status badge pills remain visible.

1.9 WHEN a page containing the "Graduate Attributes / Historical Evidence / Competency Frameworks" cards renders its area below the primary content THEN the system displays a large solid black region as the bottom of the page, and an onboarding checklist widget in the bottom-left of that region is barely visible.

1.10 WHEN the application renders in dark mode across any role (Admin, Coordinator, Teacher, Student, Parent) THEN the system renders primary CTAs with black text on the brand teal→blue gradient instead of white text, card body text becomes invisible because foreground color tokens do not swap against dark backgrounds, and the brand gradient appears inconsistent across surfaces (correct on some buttons and wrong on others), producing an overall "bad" dark-mode appearance that the user has explicitly called out as broken.

1.11 WHEN a new user visits the `/login` page THEN the system displays only the existing "Sign in" flow and does not render any "Create account", "Sign up", or "Register" link or button, so there is no UI path from the login page to a self-registration flow.

1.12 WHEN an authenticated user is on any role dashboard (Admin, Coordinator, Teacher, Student, Parent) THEN the system does not render any sign-out button, link, or menu item anywhere in the dashboard chrome (sidebar, header, or profile area), so the user cannot sign out from the UI.

1.13 WHEN an authenticated user views the top header of any role layout (AdminLayout, CoordinatorLayout, TeacherLayout, StudentLayout, ParentLayout) THEN the system renders the language-change control but does not render a profile control (avatar + name + dropdown) immediately to the right of the language-change button, so there is no discoverable entry point to view the profile or to trigger sign-out from the top bar.

1.14 WHEN a student user signs in for the first time THEN the system routes them into the onboarding wizard but opens the wizard at a later step (observed: step 4 / `baseline_select`) instead of step 1 (`welcome`), because the wizard reads a persisted `current_step` value from onboarding progress that was pre-seeded to a mid-flow step, and there is no first-time-login check that distinguishes a genuinely-new user from a resuming user.

1.15 WHEN any user (Admin, Coordinator, Teacher, Student, Parent) opens their dashboard for the first time THEN the system does not launch a guided tour; instead, a "Quick Start" checklist widget is rendered at the bottom of the dashboard, which is inconspicuous, uncontextualized, and easy to miss, so first-time users have no guided walkthrough of the platform's features.

1.16 WHEN an authenticated user views the top header of any role layout THEN the system does not render a dedicated Settings icon button at the rightmost position of the top bar; settings are accessible only via sidebar/menu navigation, with no consistent top-bar affordance across roles.

1.17 WHEN an admin clicks the "Remind" button on a row of the Pending Onboarding page THEN the system invokes the `send-onboarding-reminder` Edge Function but there is no end-to-end verification that the button correctly shows loading state, displays a success toast with the recipient's email on 2xx responses, displays an error toast with the backend's error message on non-2xx responses, disables itself while the call is in flight to prevent double-submits, and is idempotent (clicking twice within a short window does not send two emails).

1.18 WHEN a user (any role) views their profile page THEN the system does not expose an "Upload profile picture" / "Change avatar" action; existing `profiles.avatar_url` (if set) is not consistently rendered; sidebar and top-bar profile controls show placeholder initials or nothing instead of the user's uploaded avatar.

1.19 WHEN a user submits a form that violates a database CHECK/UNIQUE/FOREIGN KEY/NOT NULL constraint (observed example: creating a Badge Spotlight schedule with a non-Monday `week_start` triggers `new row for relation "badge_spotlight_schedule" violates check constraint "week_start_is_monday"`) THEN the system surfaces the raw PostgreSQL error message to the user (or a generic failure toast), with no field-level explanation of what rule was violated, no prevention of the error at the form-input level, and no consistent pattern applied across the platform.

1.20 WHEN a student clicks the "Complete & Go to Dashboard" button at the end of the onboarding wizard THEN the system waits synchronously for the `process_onboarding` Edge Function round-trip (score recomputation, personality/self-efficacy/learning-style/study-strategy write-back, AuthProvider profile re-fetch) before navigating, and during that wait (observed 2–10 seconds) the UI gives no feedback — the button does not show a spinner, the page does not show a loader/skeleton, and no optimistic navigation occurs — so the user perceives the application as hung; compounding the delay, once the Student dashboard finally mounts it fires many parallel queries on first render (courses, ILO attainment, XP history, habits heatmap, leaderboard, realtime channel) some of which block the first paint rather than rendering per-section skeletons.

1.21 WHEN an authenticated user on the Admin, Coordinator, Teacher, or Parent role loads their dashboard THEN the system renders the dashboard directly into its KPI row / section cards with no personalized greeting hero at the top, so only the Student dashboard has a "Good morning, {name}" hero while every other role dashboard opens cold with no welcome surface, producing inconsistent cross-role first impressions.

1.22 WHEN any list, grid, or table view across the platform renders with zero items (for example the Courses page with no enrolled courses, the Progress page with no progress entries, ILO/PLO/CLO lists, Assignments lists, the Users table, the Notifications feed, Leaderboard, Badges grid, Marketplace, Evidence feed, Audit Log, Pending Onboarding, or Parent Linked Students) THEN the system renders a completely blank content area with no empty-state illustration, no explanatory copy describing what the surface is, no guidance for what the user can do next, and no contextual CTA.

1.23 WHEN the Student dashboard renders the "All habits" habit tracker card's color-gradient legend beneath the heatmap grid THEN the system clips the legend on the left edge — one or more of the leading color swatches (from lightest `bg-slate-200` through the mid-tones to darkest `bg-red-500`) are partially or fully hidden by the card's `overflow-hidden` or by misaligned content padding, and the "4+ habits" caption sits at the right visually detached from the swatches that should immediately precede it, so the legend is neither fully visible nor readable.

1.24 WHEN the user moves the cursor across adjacent cells of the habit heatmap grid THEN the system re-renders the tooltip on every `mousemove`, the cell hover state flickers on and off, and the tooltip repositions abruptly with no smooth open/close animation between adjacent cells; cells have no `transition-transform` on the hover state so there is no gentle zoom, and the tooltip primitive is not configured with `delayDuration` / `skipDelayDuration` to carry the open state across nearby cells, making the interaction feel laggy, glitchy, and broken.

1.25 WHEN the user toggles a switch control that is wired to a backend mutation (for example the Wellness tracking opt-in switch or the Public profile switch) from off to on or vice versa THEN the system delays the visual state change until after the server round-trip completes — the `checked` prop is bound directly to the mutation result with no optimistic update — so the Radix Switch thumb does not slide smoothly in response to the click, the track color change is abrupt or delayed, and the user perceives the toggle as slow, unresponsive, and unanimated.

1.26 WHEN the user switches the application locale to Arabic THEN the system flips layout direction to RTL but leaves many user-facing strings in English across multiple pages and roles: page titles, section headings, subheadings, button labels, menu items, form field labels, empty-state titles / descriptions / CTAs, Sonner toast messages, dialog titles and confirmation prompts, DataTable column headers, badge labels and status pills, and error messages returned by `mapSupabaseError` remain untranslated; date/number formatting does not adopt the `ar-QA` Intl locale consistently, Arabic-Indic numerals and Arabic month names are inconsistent, and OBE/accreditation terminology (ILOs, PLOs, CLOs, Bloom's Taxonomy levels, assessment, curriculum, accreditation, CQI) is not aligned with formal Qatar Ministry of Education and Higher Education (MOEHE) conventions in Modern Standard Arabic (MSA).

1.27 WHEN any authenticated role (Admin, Coordinator, Teacher, Student, Parent) views its layout THEN the system does NOT render a single full-width horizontal top header that spans the entire viewport from the left edge to the right edge; instead (a) the logo lives inside the fixed `w-64` sidebar and the top header starts to the right of the sidebar, breaking the expected product-chrome convention where the brand mark is the leftmost element of the global header bar, (b) the header content (search, language switcher, bell, settings gear, profile dropdown) is constrained inside a `max-w-7xl mx-auto` wrapper so on wide displays the right-cluster controls float away from the viewport's right edge leaving large ambient whitespace, and (c) the per-role primary sub-navigation (e.g., Dashboard, Courses, Assignments, Grading, Users, Matrix, Linked Students) is rendered inside a separate sidebar column rather than as a secondary row directly below the global header, so the two-level header hierarchy (logo + identity on top, primary nav below) that the user expects is not present.

1.28 WHEN the top header renders THEN the system renders a dedicated Settings gear icon button (clause 1.16 / 2.16) as a second settings entry point even though the existing profile dropdown (clause 2.13) already exposes equivalent Settings / Profile Settings entries, producing a duplicated affordance that clutters the top-bar right cluster, splits the mental model of "where do I go to configure things" across two controls, and wastes valuable right-cluster horizontal space that could otherwise accommodate the unified profile menu and the notification bell.

1.29 WHEN the notification bell icon in the top header renders for any role (Admin, Coordinator, Teacher, Student, Parent) THEN the system displays the bell as a decorative static icon with no connection to the backend: (a) it is not wired to the `useUnreadCount(userId)` TanStack Query hook so the bell never shows the authenticated user's unread count badge, (b) it does not subscribe to Supabase Realtime `postgres_changes` events on `public.notifications WHERE user_id = auth.uid()` so newly-inserted notifications never update the bell in real time, (c) clicking the bell does not open a role-aware notifications panel that lists the user's notifications filtered by the per-role notification types they are entitled to receive, and (d) the underlying `notifications` table + RLS + Edge-Function-generator pipeline that would populate role-appropriate notifications (student: `grade_released`, `new_assignment`, `badge_earned`, `streak_at_risk`, `peer_milestone`, `perfect_day_nudge`, `prerequisite_unlocked`, `digest`; teacher: `grading_queue_new`, `course_announcement_required`, `at_risk_student_flagged`, `digest`; coordinator: `cqi_plan_review_due`, `curriculum_gap_flagged`, `outcome_attainment_drop`, `digest`; admin: `pending_approval`, `rate_limit_exceeded`, `system_alert`, `user_onboarded`, `digest`; parent: `child_grade_released`, `child_at_risk`, `child_streak_milestone`, `fee_due`, `digest`) is not enforced end-to-end, so even if a notification exists in the database, (i) RLS may expose it to the wrong role or filter it from the right role, (ii) no cron/edge-function generator produces role-appropriate notifications on the documented trigger events, and (iii) the in-app bell is the wrong surface for the wrong audience.

1.30 WHEN the user switches the locale to Arabic (RTL) THEN the top header logo-leftmost / primary-nav-below structure (from clause 1.27) does NOT mirror correctly: the logo remains visually leftmost (which is correct for an LTR locale but wrong in RTL, where the logo should be at the start — i.e. the right edge — per logical-property layout rules), the search input sits on the LTR-physical right instead of on the RTL-start edge, and the primary-nav row does NOT reverse its item order so "Dashboard" is not at the RTL start while the last nav item is not at the RTL end, breaking the mirrored header semantics across locales.

1.31 WHEN the sidebar renders on the Admin role layout (and the same pattern on Coordinator, Teacher, Student, Parent layouts where applicable) THEN the system renders a "Profile" `NavLink` item inside the sidebar navigation (observed: `AdminLayout.tsx` nav array contains `{ to: "/admin/settings/profile", icon: UserCircle, label: "Profile" }` as the last item) even though the top-bar profile dropdown (clause 1.13 / 2.13) already exposes a "My Profile" entry pointing at the same route; the duplication means the same destination is reachable from two different surfaces in the same chrome, splits user attention, wastes vertical sidebar space that could be used for primary role-specific nav, and violates the single-entry-point convention applied to Settings in clause 2.28.

1.32 WHEN the Admin dashboard renders its content area and the content is shorter than the viewport (for example on a wide display where the "Graduate Attributes / Historical Evidence / Competency Frameworks" three-card row sits well above the viewport bottom) THEN the system shows a large dark / near-black strip underneath the last row of content — a concrete instance of the global defect captured in clause 1.9, with the Admin-dashboard-specific root cause narrowed to (a) `<main className="flex-1 overflow-auto bg-slate-50">` in `AdminLayout.tsx` is sized to `h-screen` via `flex-1`, but when the user scrolls the main's inner `<div className="p-6">` past the bottom of the viewport on certain mobile / zoom configurations the browser shows the ancestor `<body>` / `<html>` background (which is the Tailwind default of transparent, so the OS or browser dark-background bleeds through), (b) a stale `.dark` class may remain on `<html>` from an earlier theme toggle so descendant nodes that use unresolved HSL variables (e.g. `bg-background` where `--background` is set to the dark token) render as near-black under what should be a light-mode surface, and (c) any lingering hard-coded `bg-slate-900` / `bg-gray-900` / `bg-black` utility on an orphan wrapper further down the dashboard would compound the effect. The observable outcome is that the Admin dashboard appears to "run out" into a dark void below its content, breaking the platform-standard light-mode page composition (`bg-slate-50` from the layout's main) that every other dashboard respects.

### Expected Behavior (Correct)

Each expected-behavior clause pairs with the current-behavior clause of the same index.

2.1 WHEN the admin dashboard renders the "Schedule" panel and the "Bonus Categories" grid THEN the system SHALL display headings and body text using the design-system typography tokens with accessible contrast (headings in dark slate/gray via the documented page-title and section-heading tokens, body text in `text-sm font-medium` at gray-700 or darker) on the white card background, so all labels, dates, categories, and values are legible without interaction.

2.2 WHEN a primary CTA button renders anywhere in the platform THEN the system SHALL apply the brand gradient `linear-gradient(93.65deg, #14B8A6 5.37%, #0382BD 78.89%)` (or the Tailwind-equivalent `bg-gradient-to-r from-teal-500 to-blue-600` utility) consistently, and no non-brand gradient shall be used for primary CTAs across any role.

2.3 WHEN a section Card with a brand gradient header renders THEN the system SHALL apply `overflow-hidden` on the Card and place the gradient header as the first direct child with `px-6 py-4` and no wrapping padding or margin, so the gradient reaches the top, left, and right edges of the Card with no white strip.

2.4 WHEN a KPI card has no data to display THEN the system SHALL render a professional empty-state representation (for example "0 / 0", "— / —", or "No data yet") using the same `text-2xl font-black` value typography and layout as populated KPI cards, consistent across every dashboard.

2.5 WHEN a modal, dialog, sheet, popover, or dropdown menu opens THEN the system SHALL render its surface with a white background and dark text matching the platform's card styling (`bg-white` with dark slate/gray body text), with no black or dark default surface on Shadcn Dialog, Sheet, Popover, or DropdownMenu primitives.

2.6 WHEN the Bonus XP and Badge Spotlight pages render their section headings THEN the system SHALL display heading text with visible contrast, using the standard section-card gradient header pattern (white text on brand gradient) or dark text on white as appropriate, so every section heading is legible without interaction.

2.7 WHEN any table renders its column header row THEN the system SHALL display header labels with visible contrast (dark text on a neutral header background per the shared `DataTable` convention) at all times, independent of hover or focus state.

2.8 WHEN the Users page (and every data table across every role) renders its rows THEN the system SHALL display all text cells (names, emails, arbitrary text columns) with visible contrast against the row background, matching the design-system body text token, so no row renders with invisible non-badge content.

2.9 WHEN any page renders content below its primary content area THEN the system SHALL NOT display a large solid black or dark wrapper surrounding unrelated content; the dark hero gradient is reserved for explicitly-designed hero cards (Student/Parent welcome hero), and page-level backgrounds SHALL remain `bg-slate-50` or `bg-white` per the sidebar layout pattern.

2.10 WHEN the application renders in dark mode across any role (Admin, Coordinator, Teacher, Student, Parent) THEN the system SHALL provide a polished dark theme of parity quality with the light theme, with no option to disable dark mode as a fallback; the only acceptable outcome is that dark mode is fully supported with correct design-token swaps. Specifically: surfaces (card, popover, dialog, sheet backgrounds) SHALL swap to the dark-mode `--card`, `--popover` tokens rather than remaining white or rendering as pitch-black; foreground text tokens (`--foreground`, `--card-foreground`, `--popover-foreground`, `--muted-foreground`) SHALL swap so all body text and labels remain legible on dark surfaces; borders SHALL use the dark `--border` token; muted, accent, primary, and destructive semantic colors SHALL each have an explicit `.dark` variant with accessible contrast against the dark backgrounds; the brand teal→blue gradient SHALL continue to render white text on all CTAs regardless of theme (never black text); the dark hero gradient `linear-gradient(135deg, #0f172a 0%, #1e3a8a 50%, #312e81 100%)` SHALL remain reserved for hero cards only and SHALL NOT leak onto section cards, data tables, or page backgrounds in dark mode. Implementation SHALL use HSL CSS variables with `:root` (light values) and `.dark` (dark values) swaps wired through Tailwind v4's `@custom-variant dark (&:where(.dark, .dark *))` directive and shadcn's standard theming so every Shadcn component picks up the swap automatically.

2.11 WHEN a new user visits the `/login` page THEN the system SHALL render a clearly visible "Create account" / "Sign up" link or button that routes to a functional self-registration flow; the self-registration flow SHALL create a new Supabase auth user and a corresponding `profiles` row respecting the existing institution-scoping and role model (the specific default role and approval semantics are a design decision deferred to design.md). This is captured as a bug because the user experiences the absence of a documented, expected-to-exist UI affordance as a defect.

2.12 WHEN an authenticated user is on any role dashboard (Admin, Coordinator, Teacher, Student, Parent) THEN the system SHALL expose a clearly discoverable UI control to sign out; activating that control SHALL invoke `supabase.auth.signOut()` and redirect to `/login`. Per clause 2.13 this control SHALL live inside the top-bar profile dropdown. This is captured as a bug because the user experiences the absence of a documented, expected-to-exist UI affordance as a defect.

2.13 WHEN an authenticated user views the top header of any role layout (AdminLayout, CoordinatorLayout, TeacherLayout, StudentLayout, ParentLayout) THEN the system SHALL render a profile section (avatar + name + dropdown) positioned immediately to the right of the existing language-change control, and the profile dropdown SHALL contain at minimum a "My Profile" link that routes to a view/edit profile page and a "Sign out" action that fulfills clause 2.12; the top-bar profile placement and dropdown contents SHALL be consistent across all five role layouts.

2.14 WHEN a student user signs in for the first time THEN the system SHALL open the onboarding wizard at step 1 (`welcome`) and SHALL NOT skip ahead to any later step; the wizard's saved-progress resume behavior SHALL apply only when the student has genuinely progressed past step 1 (meaning `current_step` has been advanced by an authenticated action, not by seed data). Demo seed data SHALL leave student onboarding at `current_step = 'welcome'` with all completion flags false.

2.15 WHEN any user opens their dashboard for the first time after account creation THEN the system SHALL launch a role-specific guided product tour as a modal/overlay flow that steps the user through the key areas of their dashboard (sidebar navigation, KPI cards, primary actions, profile/language/settings controls, any role-specific sections like Gamification for Student, Curriculum Matrix for Coordinator, Grading Queue for Teacher, Linked Students for Parent, User Management for Admin); the tour SHALL be dismissible, resumable, and re-launchable from a "Take the tour" action in the profile dropdown; the tour progress SHALL be persisted per user (not per device) so dismissal carries across browsers; the bottom-of-page Quick Start widget SHALL be removed or repurposed as the re-launch control only. The tour SHALL render consistently in both light and dark modes and respect RTL.

2.16 WHEN an authenticated user views the top header of any role layout (AdminLayout, CoordinatorLayout, TeacherLayout, StudentLayout, ParentLayout) THEN the system SHALL render a dedicated Settings icon button (Lucide Settings icon, `h-5 w-5`) as the rightmost control in the top bar, positioned after the profile dropdown from clause 2.13; clicking the Settings icon SHALL route to the role-appropriate settings page; the final top-bar ordering from left to right (in LTR) SHALL be: [existing left-side content / page title / breadcrumb] … [language switcher] [profile dropdown] [Settings icon], and SHALL mirror under RTL.

## Notes for Design Phase

The following items are recorded here so they are not lost, but they are intentionally NOT part of the bug condition C(X) and therefore NOT covered by the Fix Checking or Preservation Checking properties. They belong to design.md and tasks.md as follow-up work.

- Demo data seeding across every section of the database. The user has asked that every section of the database be seeded with realistic demo data so each UI screen populates with representative content, enabling visual verification that backend tables are correctly wired to the frontend. Implementation should be handled as Edge Function seed scripts or SQL seed migrations that use the existing tables (`profiles`, `institutions`, `programs`, `courses`, ILO/PLO/CLO and `outcome_mappings`, `assignments`, `evidence`, `xp_transactions`, `badges`, etc.), respect RLS policies, and are idempotent so they can be re-run safely. This is a separate deliverable and SHALL NOT expand the bug condition C(X) for this bugfix.

The following best-practice recommendations are derived from senior-developer web research and are non-binding suggestions to inform design.md. They are NOT bug clauses.

- **Guided tour library**: react-joyride (`gilbarbara/react-joyride`) is the mature, actively-maintained React-specific option; supports per-role step arrays, `stepIndex`-controlled mode, localStorage + server-side progress gating, and callback-based state sync. Known integration pitfall: Radix Dialog/Sheet closes on outside click during a tour step; mitigate via a Zustand `tourActive` flag that sets `modal={false}` and prevents `onInteractOutside` when the tour is running.
- **Tour progress persistence**: store per-user tour state in a new `profiles.tour_completed_at timestamptz` column (or a `user_tour_progress` table keyed by `user_id + role`) so dismissal carries across browsers and devices, not just localStorage.
- **Dark mode**: Tailwind v4 + shadcn/ui standard pattern. Use `@custom-variant dark (&:where(.dark, .dark *));` in the root CSS; define HSL CSS variables for `--background`, `--foreground`, `--primary`, `--primary-foreground`, `--muted`, `--muted-foreground`, `--card`, `--card-foreground`, `--popover`, `--popover-foreground`, `--border`, `--input`, `--ring`, `--destructive`, `--destructive-foreground`, etc. on `:root` (light values) and on `.dark` (dark values). Shadcn components already consume these variables. Ensure the brand gradient button uses a hard-coded `text-white` (not a theme variable) so the CTA text stays white in both themes. Use `next-themes`-style logic ported to a Zustand slice (or a small `useTheme` hook) for mount-safe theme detection and persistence.
- **Signup trigger**: create a Supabase `handle_new_user()` function triggered `AFTER INSERT ON auth.users FOR EACH ROW` that inserts a matching `public.profiles` row with default role (`student`) and the user's email; the function is `SECURITY DEFINER` so it bypasses RLS without exposing the service role key to the client. This avoids the "new row violates RLS policy for table profiles" failure that occurs when signup tries to insert the profile from the client session.
- **Avatar uploads**: use a public `avatars` bucket with RLS policies that allow authenticated users to INSERT/UPDATE/DELETE only their own object path (`bucket_id = 'avatars' AND (storage.foldername(name))[1] = auth.uid()::text`). On the client, use `browser-image-compression` or an `<img>` canvas resize to cap uploads at ~512x512 @ ~150 KB before upload. Store `profiles.avatar_url` as the public URL returned by `storage.from('avatars').getPublicUrl(path)`. Invalidate `queryKeys.user.profile` on upload success so the avatar updates across the UI without reload.
- **Error mapping utility**: create `src/lib/mapSupabaseError.ts` that receives a PostgrestError (or generic error) and returns `{ code, userMessage, fieldPath? }` based on PG error codes (23505, 23503, 23502, 23514, 42501) and constraint name lookup. Wire it into `useMutation.onError` callbacks globally so every form benefits automatically. For the specific `week_start_is_monday` constraint, the mapper returns `{ userMessage: "Week start must be a Monday", fieldPath: "week_start" }`.
- **Monday-only date picker**: use react-day-picker's `disabled={{ dayOfWeek: [0,2,3,4,5,6] }}` plus a Zod refinement `.refine(d => d.getDay() === 1, { message: "Must be a Monday" })` on the schema. Display the selected week as "Week of {Mar 10 – Mar 16}" using date-fns `format(start, 'MMM d')`+`' – '`+`format(addDays(start, 6), 'MMM d')`.
- **Onboarding completion performance (clause 2.20)**: implement optimistic navigation by calling `navigate('/student/dashboard')` BEFORE awaiting `processOnboarding.mutateAsync(...)`, then fire the mutation as a background side-effect. Surface the in-flight mutation via the TanStack Query `isMutating` signal as a small non-blocking top-bar progress indicator (Shadcn `<Progress>` or a thin animated bar) so the server work is visible without blocking. Alternative architecture worth considering: split the Edge Function into a synchronous-critical branch (write `current_step = 'complete'` + set `onboarding_completed_at`, return within ~200ms) and an asynchronous branch (recompute personality / self-efficacy / learning-style / study-strategy / baseline scores, award XP, enqueue welcome email via Resend) so the critical-path response completes in under 200ms while heavier post-processing runs on a queue.
- **Student dashboard cold-load performance (clause 2.20)**: audit every query the Student dashboard kicks off on mount. Convert independent data sources to `useQueries` with `staleTime: 60_000` per source so navigation away and back re-uses cached data. Render per-card skeletons with `<Shimmer />` (the existing shared component) rather than a single full-page loader, so individual sections reveal independently. Defer below-the-fold sections (leaderboard, badges, marketplace) to intersection-observer lazy-mount so they don't compete for the critical path. Thread it into all five role dashboard pages (`AdminDashboard.tsx`, `CoordinatorDashboard.tsx`, `TeacherDashboard.tsx`, `StudentDashboard.tsx`, `ParentDashboard.tsx`) as the first child of the page's `<div className="space-y-6">` wrapper. Nielsen-Norman / Atlassian research consistently flags duplicate settings entry points as a source of "two doors same room" cognitive load; keeping the single-surface profile dropdown is the documented pattern for SaaS admin chrome.

- **Role-based notification system end-to-end (clause 2.29)**: the pipeline composes three layers consistent with industry best practices (DesignGurus "Designing a Notification System", Velt.dev "Scalable In-App Notification Systems 2026", Suprsend "Notification System for SaaS 2026"): **event layer** — database AFTER-triggers + pg_cron jobs + Edge Functions emit row(s) into `public.notifications` keyed by `user_id`, with a role-aware `type` enum and a `metadata jsonb` carrying the CTA payload (e.g. `{ submission_id, course_id }` for `grade_released`, `{ student_id, child_name }` for `child_at_risk`); **storage / authorization layer** — RLS policy `notifications_user_read_own` enforces `user_id = auth.uid()` on SELECT, `notifications_user_update_own` enforces the same on UPDATE (for `is_read`), and INSERT/DELETE are service-role-only so clients can never self-emit; the per-role `type` values are gated by a CHECK constraint augmented by an app-layer `ROLE_TO_ALLOWED_TYPES` map so a compromised service-role token cannot emit `pending_approval` to a student; **delivery layer** — the existing `useUnreadCount` + `useNotifications` + `useMarkAsRead` hooks drive the bell badge + popover, while `useNotificationRealtime` subscribes to `postgres_changes` on `notifications` with filter `user_id=eq.{uid}` and invalidates the TanStack Query cache on every INSERT. Supabase Realtime Authorization (announced 2024-08; GA changelog referenced) SHALL NOT be strictly required because INSERT filters already narrow by `user_id`, but enabling Realtime RLS on `realtime.messages` is RECOMMENDED as defense-in-depth so a leaked WebSocket URL cannot eavesdrop on other users' notification streams. **Cron/Edge generators** — `supabase/functions/notification-digest` (daily, aggregates low-priority events into a single per-role `digest` row for users whose preferences gate individual emissions), `supabase/functions/streak-risk-notify` (clause 2.29 `streak_at_risk`), `supabase/functions/at-risk-prediction-notify` (teacher `at_risk_student_flagged` + parent `child_at_risk`), `supabase/functions/fee-due-notify` (parent `fee_due`). **Client presentation** — the bell popover renders role-aware groups: Today, Earlier this week, Older; each row shows a type-colored Lucide icon (student grade → `GraduationCap emerald-600`, at-risk → `AlertTriangle amber-600`, admin pending → `UserPlus blue-600`, parent child-milestone → `Heart pink-500`), notification title (i18n), body (i18n, supports interpolation via metadata), relative time via `formatRelativeTime`, and a primary CTA derived from the metadata → role-scoped route. Quiet hours and email-channel preferences already live in `public.notification_preferences`; the bell SHALL respect `in_app = true` for surfacing and MAY suppress the Sonner toast (not the badge) during the user's quiet window.

- **Role-aware notification type enum + RLS hardening (supporting clause 2.29)**: add to `public.notifications.type` a Postgres CHECK constraint enumerating the full union across roles; add a trigger `notifications_type_role_guard()` that on INSERT looks up `profiles.role` by `user_id` and rejects rows whose `type` is not in the role's allowed set (`ROLE_TO_ALLOWED_TYPES`). This defense-in-depth prevents any service-role code path (including third-party integrations or future Edge Functions) from seeding a notification that the role's bell would not know how to render. Migration plan: `supabase/migrations/2026<...>_notifications_role_aware_types.sql` — (a) extend CHECK constraint, (b) create the guard trigger, (c) backfill any existing rows whose type is outside the new enum to `digest` (no data loss, soft migration), (d) index `(user_id, is_read, created_at DESC)` so the bell query stays O(log n) at scale.

- **Realtime Authorization hardening (supporting clause 2.29)**: per Supabase's Realtime Authorization GA (2024-08), enabling RLS on `realtime.messages` plus a per-channel authorization policy adds a second layer of defense so even with a leaked WebSocket JWT an attacker cannot subscribe to another user's notification stream. Recommended policy: `CREATE POLICY notifications_realtime_read_own ON realtime.messages FOR SELECT TO authenticated USING (realtime.topic() = 'notifications:' || auth.uid()::text);` and corresponding client-side `supabase.channel('notifications:' || user.id, { config: { broadcast: { self: false } } })` topic naming. Adoption is OPTIONAL for this bug condition but STRONGLY RECOMMENDED before we scale beyond the current single-institution pilot.

The design phase SHALL select the specific libraries and patterns and produce a design.md that adopts or deviates from these recommendations with reasoning.
