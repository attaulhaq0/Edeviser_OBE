Every clause reference in this document (1.n / 2.n / 3.n) maps directly to the corresponding clause in `.kiro/specs/ui-consistency-global-fixes/bugfix.md`. No clause is renumbered. No requirement is re-stated that is not already locked in bugfix.md — this document is purely how.

## 1. Overview

This fix closes 26 defect groups (bugfix.md clauses 1.1–1.26) observed across the Admin dashboard and suspected across the four remaining role surfaces (Coordinator, Teacher, Student, Parent). The defects cluster into six interlocking themes: (a) design-system compliance drift (invisible text, rogue gradients, white strip above gradient section headers, black dialog surfaces, blank KPI empty states, unsafe dark surfaces under data tables), (b) broken dark mode (missing token swaps, black text on brand CTAs, inconsistent gradient rendering), (c) authentication-surface gaps (no self-registration link, no sign-out control, missing profile dropdown, missing rightmost Settings icon), (d) first-time experience (onboarding wizard opens mid-flow for genuinely-new students, Quick Start widget pinned at bottom instead of a per-role guided tour, missing greeting hero on 4 of 5 role dashboards), (e) performance of critical transitions and first-render shape (synchronous `processOnboarding` blocks Complete & Go to Dashboard for 2–10s, Student dashboard fires many parallel blocking queries on cold mount), and (f) i18n coverage + micro-interaction polish (Arabic strings missing outside the label layer, habit heatmap legend clipping, habit heatmap hover jitter, optimistic-update parity for backend-bound toggles, raw PG constraint errors surfacing to the UI, missing avatar upload, non-functional Remind button).

The single top-level goal: **bring every rendered page and interaction across every role into compliance with the existing design-system tokens, eliminate the observed UX defects, and close the auth + onboarding + Arabic coverage gaps — all without regressing any non-buggy surface** (clauses 3.1–3.26).

## 2. Bug Condition C(X), Fix Checking FC, Preservation Checking PC

Let `X` be the full rendered UI + auth/onboarding/i18n state of the Edeviser app, parameterized over the tuple `{ role ∈ {admin, coordinator, teacher, student, parent}, page ∈ all routes, theme ∈ {light, dark}, locale ∈ {en, ar-QA}, profileState ∈ {new, resuming, completed-onboarding}, dataState ∈ {empty, populated, error} }`.

### Bug Condition C(X)

C(X) holds iff **at least one** of the 26 current-behavior clauses 1.1 through 1.26 in bugfix.md is observable on the rendered surface. Formally:

```
FUNCTION isBugCondition(X)
  INPUT: X — a rendered UI state tuple (role, page, theme, locale, profileState, dataState)
  OUTPUT: boolean

  RETURN EXISTS n IN {1, 2, …, 26} SUCH THAT clause_1_n_holds(X)

END FUNCTION
```

Each `clause_1_n_holds(X)` is the concrete predicate captured in clause 1.n of bugfix.md. Examples:

- `clause_1_1_holds(X)` ↔ on the Admin dashboard, the "Schedule" panel or "Bonus Categories" grid renders text with insufficient contrast against the white card background.
- `clause_1_5_holds(X)` ↔ a Shadcn Dialog / Sheet / Popover / DropdownMenu renders with a dark surface and white text.
- `clause_1_14_holds(X)` ↔ a genuinely-new student lands on step `baseline_select` instead of `welcome`.
- `clause_1_20_holds(X)` ↔ click-to-navigation-start on Complete & Go to Dashboard exceeds 200ms or TTI of the Student dashboard exceeds 1500ms on a cold load.
- `clause_1_26_holds(X)` ↔ Arabic locale renders any JSX text node outside the tiny allowlist that is not routed through `t()`.

### Fix Checking FC

FC is the conjunction of the 26 expected-behavior predicates:

```
FUNCTION fixChecking(X)
  INPUT: X — a rendered UI state tuple with the fix applied
  OUTPUT: boolean

  RETURN FOR ALL n IN {1, 2, …, 26}: clause_2_n_holds(X)
END FUNCTION
```

In practice, clause 2.n is verified by the test harness that pairs it to clause 1.n. See §6 for the per-clause property tests.

### Preservation Checking PC

PC is the conjunction of the 26 regression-prevention predicates:

```
FUNCTION preservationChecking(X_original, X_fixed)
  INPUT: X_original — rendered state from the pre-fix build; X_fixed — rendered state from the post-fix build
  OUTPUT: boolean

  RETURN FOR ALL n IN {1, 2, …, 26}: clause_3_n_holds(X_original, X_fixed)
END FUNCTION
```

`clause_3_n_holds` requires the documented unchanged behavior to render identically before and after the fix on surfaces outside the 1.n scope.

### Pseudocode sketch of how property tests encode FC ∩ PC

```
// DOM audits (clauses 1.1–1.10, 2.1–2.10, 3.1–3.10)

    fc.record({
      role: fc.constantFrom('admin','coordinator','teacher','student','parent'),
      page: fc.constantFrom(...everyRouteUnderRole(role)),
      theme: fc.constantFrom('light','dark'),
    }),
    async ({ role, page, theme }) => {
      await mountAs(role, page, { theme });
      // FC: no prohibited rendering
      expect(scanForGradientOnCtaButtons()).toEqual(BRAND_GRADIENT);
      expect(scanForDialogSurfaces()).toEqual(WHITE_OR_CARD_TOKEN);
      expect(scanForInvisibleText()).toEqual([]);
      expect(scanForWhiteStripAboveGradientHeaders()).toEqual([]);
      // PC: non-buggy surfaces unchanged
      expect(domSnapshot(scope('non-bug'))).toMatchBaseline();
    }
  ),
  { numRuns: 100 }
);

// RLS / signup (clauses 2.11, 2.18, 3.11, 3.12, 3.18)
fc.assert(fc.property(generateSignupInputs, async (input) => {
  const { user, profile } = await supabase.auth.signUp(input);

  expect(await supabase.auth.signInWithPassword(existingUser)).toHaveProperty('data.session');
}), { numRuns: 100 });

// Performance (clause 2.20)
performance.mark('click');
await userEvent.click(screen.getByRole('button', { name: /Complete & Go to Dashboard/ }));
performance.mark('nav-start');
await screen.findByTestId('student-dashboard-root');
performance.mark('first-content');

expect(performance.getEntriesByName('click-to-nav')[0].duration).toBeLessThan(200);
expect(performance.getEntriesByName('click-to-ttfc')[0].duration).toBeLessThan(800);

// i18n coverage (clause 2.26)
const hardCodedEnglishHits = astScan(['src/pages/**/*.tsx','src/components/**/*.tsx'], {
  rule: 'jsx-text-without-t-call',
  allowlist: ['MB','KB','%','Edeviser','ILO','PLO','CLO'],
});
expect(hardCodedEnglishHits).toEqual([]);
```

The three checks together (plus the preservation baseline under `audit/baselines/`) give us mechanical coverage of C(X), FC, and PC without human visual inspection.

## 3. Architecture decisions (ADRs)

Each ADR: decision → alternatives considered → reasoning → regression clauses respected.

### ADR-01 — Dark mode via HSL CSS variables + Tailwind v4 `@custom-variant dark` + shadcn tokens + Zustand `useTheme` slice

**Decision.** Ship a polished dark theme by extending `src/index.css` with full HSL variable sets on `:root` (light) and `.dark` (dark), driven by a single Zustand `themeStore` with a `themeMode: 'light' | 'dark' | 'system'` field (persisted to `profiles.theme_preference` when authenticated, to `localStorage` otherwise). The brand gradient CTA text remains hard-coded `text-white` regardless of theme.
**Alternatives considered.** `next-themes` (chosen against: Vite, not Next.js); CSS `color-scheme` only (insufficient for Shadcn token swap); disabling dark mode (explicitly rejected by the user in clause 1.10 / 2.10).
**Reasoning.** Tailwind v4's `@custom-variant dark (&:is(.dark *))` is already in `src/index.css`; Shadcn components already consume the HSL variables; the missing pieces are: (a) dark variants for `--primary-foreground`, `--muted-foreground`, `--popover-foreground`, `--card-foreground` that currently leak white-on-white on some surfaces under `.dark`, and (b) a theme-toggle UI hooked to the `profiles` row.
**Regression clauses respected.** 3.1, 3.4, 3.7, 3.8, 3.9, 3.13 (Shadcn primitives' accessibility behavior untouched; dark hero gradient reserved for hero cards only).

### ADR-02 — Guided tour via react-joyride + Zustand `tourActive` + per-user `profiles.tour_completed_at`

**Decision.** Adopt `react-joyride` (wrapped in `src/components/shared/GuidedTour.tsx`) with per-role step arrays in `src/lib/tours/{role}Tour.ts`. A `tourStore` Zustand slice exposes `{ tourActive, currentRoleTourId, start, skip, complete }`. Persistence lives in `profiles.tour_completed_at` (per-user, cross-device). The existing `QuickStartChecklist.tsx` is deprecated then removed.
**Alternatives considered.** `driver.js` (lighter but lacks React bindings for step callbacks we need); building bespoke tooltips (duplicates react-joyride capabilities, no gain).
**Reasoning.** Radix Dialog/Sheet closes on outside click during a tour step; the `tourActive` flag is read by a small helper that sets `modal={false}` and suppresses `onInteractOutside` while the tour runs. Re-launch lives in the profile dropdown ("Take the tour").
**Regression clauses respected.** 3.9, 3.15 (users who have already completed or dismissed the tour never see it again), 3.13.

### ADR-03 — Signup via Supabase client `signUp()` + `handle_new_user()` SECURITY DEFINER trigger

**Decision.** Expose self-registration at `/signup` using `supabase.auth.signUp(email, password, { data: { full_name, institution_id } })`. A new `handle_new_user()` Postgres function marked `SECURITY DEFINER` runs `AFTER INSERT ON auth.users` and inserts a matching `public.profiles` row (default role: `student`, `institution_id` from signUp metadata, `tour_completed_at = null`, `theme_preference = 'system'`).
**Alternatives considered.** Inserting `profiles` from the client (rejected — fails the existing profiles RLS policy with "new row violates RLS policy"); routing signup through an Edge Function (rejected — heavier path for what is essentially one INSERT).
**Reasoning.** SECURITY DEFINER + `search_path = public` is the documented Supabase pattern and stays compatible with our RLS helper functions (`auth_user_role()`, `auth_institution_id()`).
**Regression clauses respected.** 3.11 (existing sign-in form and redirect untouched), 3.12 (existing auth wiring reused, no parallel flow).

### ADR-04 — Avatar upload via public `avatars` bucket + client-side resize + folder-scoped RLS

**Decision.** Reuse the existing `avatars` bucket (already provisioned in migration `20260416193532_create_storage_buckets.sql`). Add a shared `AvatarUpload` component that (a) validates type/size via Zod, (b) resizes to ≤512×512 @ ≤150 KB via `browser-image-compression`, (c) uploads to `{user_id}/{uuid}.{ext}`, (d) updates `profiles.avatar_url`, (e) invalidates `queryKeys.user.profile`. Migration `…_tighten_avatars_bucket_policies.sql` ensures the final policy set is INSERT/UPDATE/DELETE folder-scoped + SELECT authenticated.
**Alternatives considered.** Server-side resize via an Edge Function (adds latency and a deploy dependency for a simple client transform); a private bucket with signed URLs (overkill for avatars, adds URL-refresh complexity).
**Reasoning.** Buckets exist; what is missing is (a) the client-side resize pipeline and (b) tightening policies so users can only write their own folder. The existing `20260504033018_restrict_avatars_bucket_listing.sql` already scopes SELECT; we extend that pattern to INSERT/UPDATE/DELETE.
**Regression clauses respected.** 3.18 (users without an uploaded avatar continue to see initials), 3.9.

### ADR-05 — Centralized error mapping via `src/lib/mapSupabaseError.ts` + `useStandardMutation` wrapper

**Decision.** Create `mapSupabaseError(err): { code: string; userMessage: string; fieldPath?: string }` that switches on PG error codes (23505, 23503, 23502, 23514, 42501) and constraint names (e.g. `week_start_is_monday → { userMessage: 'Week start must be a Monday', fieldPath: 'week_start' }`). A thin `useStandardMutation` wrapper (around TanStack's `useMutation`) applies the mapper in `onError` globally, logs the raw error to Sentry, and emits a Sonner toast with the user message.
**Alternatives considered.** Ad-hoc mapping at each call site (rejected — would keep drift alive); a Postgres `hint`/`detail` parser (rejected — fragile).
**Reasoning.** One file owns the mapping. Every mutation hook can adopt `useStandardMutation` as a drop-in replacement over time; existing hooks are backward-compatible until migrated.
**Regression clauses respected.** 3.19 (success toasts unchanged), 3.9.

### ADR-06 — Optimistic onboarding-complete navigation

**Decision.** On Complete & Go to Dashboard, call `navigate('/student/dashboard')` **before** awaiting `processOnboarding.mutateAsync(...)`. The mutation runs fire-and-forget, surfaced via `queryClient.isMutating({ mutationKey: ['processOnboarding'] })` as a non-blocking top-bar Shadcn `<Progress>` indicator. On server failure the wrapper emits a mapped Sonner toast with a Retry action.
**Alternatives considered.** Splitting the Edge Function into synchronous-critical + asynchronous branches (documented as an open follow-up; heavier change, not needed to hit the 200ms SLO); preloading the Student dashboard route chunk during the wizard (orthogonal optimization, worth keeping but not blocking).
**Reasoning.** The ~2–10s delay is latency in `process_onboarding`, not in the navigation. Decoupling is sufficient to satisfy clause 2.20's 200ms click-to-nav-start SLO without changing server-side persistence semantics (clause 3.20).
**Regression clauses respected.** 3.20 (server-side processing unchanged, schema writes identical), 3.9.

### ADR-07 — Universal `WelcomeHero` via `src/components/shared/WelcomeHero.tsx`

**Decision.** Extract a single `WelcomeHero` consuming `{ name, role, subtitle, stats? }` and render it as the first child of every role dashboard's `<div className="space-y-6">` wrapper. The Student variant preserves its existing content by threading its XP / Level / Streak chips in via the `stats` slot.
**Alternatives considered.** Per-role hero components (rejected — duplicates the gradient / greeting logic five times; makes preservation of the Student variant harder to verify).
**Reasoning.** One component, one gradient, one greeting-by-hour computation via `new Date().getHours()` + `Intl.DateTimeFormat().resolvedOptions().timeZone`. Subtitle and stats come in as props, keeping per-role copy locale-aware via `t()`.
**Regression clauses respected.** 3.21 (Student hero output unchanged by migration), 3.9.

### ADR-08 — `EmptyState` coverage via the extended shared component

**Decision.** Extend `src/components/shared/EmptyState.tsx` to the documented shape `{ icon: LucideIcon; title: string; description: string; cta?: { label: string; onClick: () => void } }`. Provide a small library of named variants (`no-courses`, `no-progress`, `no-users`, `no-notifications`, `no-badges`, `no-evidence`, `no-marketplace-items`, `no-linked-students`) that pre-fill icon / title / description copy.
**Alternatives considered.** Per-page empty states (rejected — duplicates 13+ times); generic "Nothing here yet" text (rejected — no actionable CTA).
**Reasoning.** One component, one surface specification, one property test that asserts every listed page renders an `<EmptyState>` when `data?.length === 0`.
**Regression clauses respected.** 3.22 (populated lists unaffected; empty-state branch only fires when the data array is empty after loading resolves).

### ADR-09 — Heatmap smoothness via Radix Tooltip + `TooltipProvider` + `React.memo` cells

**Decision.** Wrap the entire heatmap in a single `<TooltipProvider delayDuration={200} skipDelayDuration={100}>` and replace any inline tooltip with Shadcn `Tooltip`. Cell component is wrapped in `React.memo` with a stable `${year}-${month}-${day}` key and `transition-transform duration-150 hover:scale-110`. Reduced motion disables the zoom via a CSS media query.
**Alternatives considered.** Replacing the heatmap with a canvas-based renderer (overkill; breaks accessibility); debouncing the tooltip per cell (adds lag without fixing re-mount flicker).
**Reasoning.** `skipDelayDuration` is the specific Radix feature that carries tooltip open state across rapid cursor movement between adjacent cells; `React.memo` prevents the entire grid re-rendering on unrelated parent-state changes.
**Regression clauses respected.** 3.23 (heatmap data/color computation unchanged), 3.24.

### ADR-10 — Optimistic toggles via `src/hooks/useOptimisticToggle.ts`

**Decision.** One reusable hook: `useOptimisticToggle<T>({ queryKey, mutationFn, field })`. Implements the standard TanStack Query `onMutate` → `onError` rollback → `onSettled` invalidate pattern, surfaces a small `isOptimistic` flag for the caller to render an adjacent spinner without disabling the switch.
**Alternatives considered.** Inline `onMutate` per call site (rejected — repeated 4+ times already, will proliferate); a global mutation proxy (too invasive for a narrow concern).
**Reasoning.** Switches bound to mutations are a clearly-shaped pattern; encapsulating the contract prevents future drift.
**Regression clauses respected.** 3.25 (pure client-side toggles keep synchronous local-state updates).

### ADR-11 — i18n coverage gate via i18next-cli + AST-based property test + Qatar MOEHE glossary

**Decision.** Add `i18next-cli.config.ts` at the repo root to extract every `t()` call into `src/locales/{lng}/{ns}.json` and reconcile against `ar.json`. A property test under `src/__tests__/properties/i18nCoverage.property.test.ts` AST-scans `src/pages/**` and `src/components/**` for JSX text nodes that don't pass through `t()`, with a tiny allowlist (library names, URLs, unit labels). The MOEHE glossary lives at `src/locales/ar/glossary.md`.
**Alternatives considered.** Manual translation sweep (rejected — no gate against regression); `i18next-scanner` (equivalent; chose `i18next-cli` because it has a more modern CLI and AST plugin architecture).
**Reasoning.** Without a gate, coverage drifts within one PR. With a gate, any new JSX text either passes through `t()` or fails CI.
**Regression clauses respected.** 3.26 (existing Arabic strings preserved verbatim; the gate only fills gaps).

### ADR-12 — Invite-token system for non-student onboarding at institutional scale

**Decision.** Self-service signup at `/signup` creates `profiles.role = 'student'` by default (public tier). Teachers, Coordinators, Admins, and Parents onboard via a new `invitations` table keyed by `(institution_id, email, role)` with a signed token, 7-day expiry, single-use. An invite route `/accept-invite/{token}` validates the token server-side, reads the target role and `institution_id`, and creates the profile with the invited role instead of the default. An admin "Invite Users" page (`src/pages/admin/users/InviteUsersPage.tsx`) lets admins bulk-invite by CSV with role + institution. The companion migration `20260901000005_create_invitations_table.sql` defines the table, its RLS policies (admin-insert scoped to `institution_id = auth_institution_id()`; public-select-by-token via a `SECURITY DEFINER` function that returns invitation-by-token without exposing the table to anon), and a `send-invitation-email` Edge Function that dispatches email through Resend.
**Alternatives considered.** Promoting students to higher roles from the admin UI only (rejected — requires every teacher to first self-register as a student, which is demeaning and doesn't scale for 50+ teachers per institution); centrally-controlled admin-creates-every-account flow (rejected — doesn't scale beyond the pilot and concentrates onboarding load on admin); magic-link-only signup (rejected — loses the per-role pre-resolution that makes invite tokens the scalable path).
**Reasoning.** At scale (3+ institutions × 500+ users each), admins cannot hand-create every non-student account, and students must not be able to self-claim staff roles. A signed invite token with `(institution_id, email, role, expires_at, used_at, created_by)` is the standard pattern for enterprise SaaS onboarding, gives admins full auditability, rate-limits well (ADR-16), and keeps the default `/signup` path unambiguously student-only.
**Regression clauses respected.** 3.11, 3.12 (existing sign-in and sign-out wiring untouched; invites are strictly additive).

### ADR-13 — Multi-institution self-serve with per-institution join modes

**Decision.** Add `institutions.slug text UNIQUE NOT NULL`, `institutions.logo_url text`, `institutions.allowed_email_domains text[]`, and `institutions.join_mode text CHECK (join_mode IN ('open','invite_only','domain_restricted')) DEFAULT 'invite_only'` so each institution chooses how it accepts new users. The `/signup` flow becomes two steps: (1) choose-institution (searchable Shadcn `Command` list populated from `institutions WHERE is_active = true` via `useInstitutionBrowse`), (2) account form. `open` mode allows self-signup without an invite but sets `profiles.status = 'pending_verification'` until the email is verified (ADR-14). `invite_only` mode rejects self-signup with "This institution requires an invitation — ask your admin" and surfaces the admin contact link. `domain_restricted` mode auto-validates the email domain against `institutions.allowed_email_domains`. The extended `handle_new_user()` trigger reads `institution_id` + `role` from `raw_user_meta_data`, validates against the institution's `join_mode`, writes `profiles.status` accordingly, and inserts a `pending_approval` audit-log row for admins if the invited role is not `student`.
**Alternatives considered.** Single hard-coded institution per deployment (rejected — the user explicitly wants full scalability, not a pilot); a global "request access" queue routed to a platform super-admin (rejected — concentrates bottleneck; institutions need autonomy); domain allowlist as the only gating mechanism (rejected — Gmail/Outlook consumer domains make domain-only gating insufficient for many K-12 and private universities).
**Reasoning.** `join_mode` gives each institution autonomy over its trust boundary without forcing the platform team to pre-approve every sign-up event. The three modes cover the realistic spectrum of institutional intake policies. Storing `slug` unlocks future deep-link URLs (`edeviser.com/{slug}`), and `logo_url` + the public `institutions` select-list power the branded choose-institution step.
**Regression clauses respected.** 3.11, 3.12 (default institution behavior for existing rows: `join_mode = 'invite_only'` via column default, so no existing institution silently opens to self-signup).

### ADR-14 — Email verification required before activation for open-mode signups

**Decision.** Add `profiles.email_verified_at timestamptz` column. After signup through `join_mode = 'open'`, Supabase sends a magic-link verification email via Resend; the profile row is created with `status = 'pending_verification'` until the link is clicked (at which point `email_verified_at` is stamped and `status` flips to `active`). Dashboards render a persistent `<Banner>` instructing the user to verify if `email_verified_at IS NULL`, and access to XP earning, habit tracking, and gradebook-visible surfaces is gated on `status = 'active'`. This is folded into clause 2.11 as a scalability refinement (not a new bug clause).
**Alternatives considered.** Trusting email on first sign-in (rejected — trivial to abuse via one-shot disposable addresses at scale); SMS OTP verification (rejected — higher infra cost + regional carrier complexity + Qatar-market SMS cost model); no verification at all in open mode (rejected — floods `profiles` with zombie rows that inflate per-institution counts).
**Reasoning.** Email verification is the cheapest, most widely understood anti-spam gate, and Supabase Auth ships the primitives for it (`user.email_confirmed_at`). Pairing the confirmation with a `profiles.status` column (rather than relying solely on `auth.users.email_confirmed_at`) keeps the gating logic readable inside application SQL and RLS policies.
**Regression clauses respected.** 3.11, 3.12 (invite-only and domain-restricted modes skip the banner because their profile is created with `status = 'active'` immediately; no change for existing users).

### ADR-15 — Avatar CDN delivery via Supabase Storage + transformation URLs

**Decision.** Use Supabase Storage's built-in image transformation (`?width=200&height=200&resize=cover`) so the same uploaded avatar is served at multiple sizes (64 px for the top bar, 32 px for table rows, 128 px for the profile page) without duplicate uploads. Set `cache-control: public, max-age=31536000, immutable` on the `avatars` bucket so CDN edges cache aggressively; the file key contains a UUID so invalidation on update is a natural consequence of writing a new key.
**Alternatives considered.** Client-side resizing into multiple buckets (rejected — 3× upload bandwidth per change; ADR-04 already resizes once for storage, but the render-time variants are separate concerns); a custom Imgproxy deployment (rejected — added infra for a solved problem); re-fetching the full avatar for every surface (rejected — wasteful at 5000-user institution scale where every table row renders a row avatar).
**Reasoning.** Supabase Storage's transformation layer is included in the platform; its output is CDN-cached by default. This scales to thousands of avatar renders per page-load at near-zero marginal cost. The UUID-based key strategy (already in `AvatarUpload`) means cache busting is automatic on upload.
**Regression clauses respected.** 3.18 (users without an uploaded avatar continue to see initials; the transformation URL only applies when `profiles.avatar_url` is set).

### ADR-16 — Rate limiting for signup + invite acceptance

**Decision.** Apply three layers of abuse control: (a) 5 signup attempts / IP / hour via Supabase Auth's built-in `GOTRUE_RATE_LIMIT_EMAIL_SENT`; (b) 10 invite-acceptance attempts / IP / day enforced by an Edge-Function middleware that reads a `rate_limit_events` audit table; (c) a CAPTCHA challenge (hCaptcha or Cloudflare Turnstile) rendered on `/signup` and `/accept-invite` when the caller's IP approaches the threshold. Brute-force detection logs to `audit_logs` with `event_type = 'signup_rate_limit_exceeded'` and can optionally block the IP for 24 hours by inserting into a `blocked_ips` table referenced by the Edge Function. Migration `20260901000008_add_rate_limiting_audit.sql` adds the audit-log event-type enum values and the `blocked_ips` table.
**Alternatives considered.** No rate limiting (rejected — trivially exploitable at scale to burn email quota and spam new-user flows); per-user-only limiting (rejected — IP layer is the only signal available before an account exists); Cloudflare WAF as the only layer (rejected — couples deployment to a specific edge provider; keep the defense-in-depth in-app).
**Reasoning.** Three independent rate limits with one CAPTCHA fallback and one audit trail covers the realistic abuse surface without locking out real users during legitimate classroom onboarding bursts.
**Regression clauses respected.** 3.11, 3.12 (existing authenticated users never hit the limits because they're scoped to unauthenticated signup + invite-accept paths).

### ADR-17 — Translation memory for future languages beyond en/ar

**Decision.** The `src/locales/{lng}/` directory pattern already supports N languages. `i18next-cli.config.ts` lists `otherLngs: ['ar']` today; adding a new language only requires adding the locale JSON files under `src/locales/{lng}/` and extending the array. A translation memory file at `src/locales/_translation-memory.json` is auto-generated by `i18next-cli` on extraction and preserves past translations across refactors (renames, key moves, deletions) so translators don't re-translate unchanged copy.
**Alternatives considered.** A paid TMS integration (rejected — overkill for a two-language-today roadmap; the pattern is ready when needed); no translation memory (rejected — every refactor drops cached translations and forces costly re-translation); vendoring translations per-release (rejected — duplicates strings across versions and hides drift).
**Reasoning.** `i18next-cli` already emits translation memory; wiring the output file into the repo at a known path is a 1-line config change. Paired with ADR-11's AST coverage gate, the platform is linguistically scalable: any new language can be added in a single PR by a translator, without engineering overhead.
**Regression clauses respected.** 3.26 (existing Arabic strings preserved verbatim; translation memory is additive).

### ADR-18 — Full-width global header + primary-nav row replacing per-role sidebars

**Decision.** Ship a single `src/components/shared/GlobalHeader.tsx` two-row primitive mounted at the top of every role layout. Row 1 renders `[brand logo (logical start)] … [spacer] … [global search (md+)] [language switcher] [notification bell] [profile dropdown]`. Row 2 renders a role-aware primary-nav row fed by `src/lib/navItems.ts` (a single map keyed by role). The legacy `w-64` sidebars in `AdminLayout`, `CoordinatorLayout`, `TeacherLayout`, `StudentLayout`, `ParentLayout` are removed; each layout becomes `<> <GlobalHeader /> <main className="w-full bg-slate-50 dark:bg-background p-6"><Outlet /></main> </>`. The header is `w-full sticky top-0 z-[100] border-b border-border bg-white dark:bg-background` with **no** `max-w-*` wrapper so it truly spans the viewport. All internal spacing uses logical properties (`ms-*` / `me-*` / `ps-*` / `pe-*`) for automatic RTL mirroring (clause 2.30). The `data-tour="top-bar"`, `data-tour="primary-nav"`, `data-tour="profile"`, `data-tour="notification-bell"` anchors migrate to this component so the guided tour (clause 2.15) retargets automatically.
**Alternatives considered.** Keep the sidebar and only widen the top bar (rejected — the user explicitly asked for logo-leftmost-in-header + primary-nav-below, which is the classic two-row product chrome pattern and the sidebar duplicated that); per-role bespoke headers (rejected — creates drift across roles, which is exactly the class of bug this spec exists to fix); `Radix NavigationMenu` as the primary-nav (considered and folded in — we use its primitives for keyboard/focus semantics but wrap them in our design-system styles so the visual treatment matches the rest of the platform).
**Reasoning.** A single full-width chrome component (a) eliminates five duplicated top-bar implementations, (b) keeps the brand logo at the logical start edge where users expect it, (c) frees the horizontal space the sidebar used for content, (d) makes RTL mirroring trivial because logical properties flip automatically, and (e) centralizes the mount-point for notification bell + profile dropdown so we never have to touch five layouts to change header behavior.
**Regression clauses respected.** 3.9, 3.10, 3.13, 3.16, 3.27 (RouteGuard logic + every existing route + every existing settings page unchanged).

### ADR-19 — Single settings entry point via profile dropdown

**Decision.** Remove the standalone `SettingsIcon` button from `GlobalHeader`'s right cluster and consolidate all settings entry points into `ProfileDropdown`. The dropdown items become: My Profile → `/{role}/profile`; Profile Settings → role-scoped settings route; Institution Settings (admin only) → `/admin/settings/institution`; Theme → `ThemeToggle` submenu (clause 2.10); Take the tour → `useGuidedTour().start()` (clause 2.15); Sign out → `supabase.auth.signOut()` (clause 2.12). The `data-tour="settings"` anchor moves from the old gear button to the "Profile Settings" menu item so the guided tour still highlights the settings entry.
**Alternatives considered.** Keep both surfaces (rejected — Nielsen-Norman-style "two doors same room" cognitive load; the user explicitly called it out as clutter); remove the settings affordance entirely (rejected — users still need a discoverable settings entry point; the profile dropdown is the documented pattern for SaaS admin chrome and already contains adjacent identity controls).
**Reasoning.** One settings entry point is discoverable without being redundant. Co-locating with the user's identity controls (avatar + name) also matches the Shopify Polaris, Atlassian, and Linear patterns.
**Regression clauses respected.** 3.16, 3.28 (existing settings routes untouched; only the duplicate affordance is removed).

### ADR-20 — Role-based notification pipeline end-to-end (three-layer architecture)

**Decision.** Adopt the three-layer pattern consistent with SaaS notification best practices (DesignGurus, Velt.dev, Suprsend 2026): **(1) event layer** — database `AFTER INSERT/UPDATE` triggers on domain tables (grades, submissions, badges, audit_logs, rate_limit_events, outcome_attainment) plus pg_cron scheduled jobs (daily at 20:00 local → `streak-risk`, daily at 18:00 → `perfect-day-nudge`, daily at 09:00 → `fee-due`, weekly Sun → `notification-digest`, weekly Mon → `cqi-review-reminder`) plus Edge Functions (`at-risk-prediction-notify`, `child-milestone-fanout`) emit rows into `public.notifications`. **(2) storage + authorization** — `notifications.type` CHECK-constrained to the documented role-aware enum; a Postgres trigger `notifications_type_role_guard()` on BEFORE INSERT looks up `profiles.role` by `user_id` and rejects a row whose `type` is not in the role's allowed set. RLS: `notifications_user_read_own` (`USING user_id = auth.uid()`), `notifications_user_update_own` (`USING user_id = auth.uid()`), INSERT/DELETE restricted to service_role. Index `(user_id, is_read, created_at DESC)` so the bell query stays O(log n). **(3) delivery** — existing `useNotifications`, `useUnreadCount`, `useMarkAsRead`, `useMarkAllAsRead`, `useDeleteNotification` hooks remain the client contract (no parallel stack). `useNotificationRealtime` subscribes to `postgres_changes` on `notifications` filtered `user_id=eq.{auth.uid()}` and invalidates `queryKeys.notifications.all` on INSERT. A new `NotificationBell` component in `src/components/shared/NotificationBell.tsx` mounts inside `GlobalHeader`, reads `useUnreadCount`, renders a red badge with `1` — `9+` when > 0, and on click opens a Shadcn Popover containing the role-aware grouped list (Today / Earlier this week / Older). Row CTA is derived from `metadata` + `type` via a pure `src/lib/notificationCta.ts` function. Quiet hours from `notification_preferences` suppress the Sonner toast (not the badge). Accessibility: `aria-label` via `t()`, unread count announced via `aria-live="polite"` on the badge, popover is `role="dialog"` with focus trap.
**Alternatives considered.** Client-side polling only (rejected — doesn't meet real-time expectation, adds Supabase load); a single Edge Function "notification dispatcher" receiving every event (rejected — spreads domain logic off-schema and adds a cold-start penalty for every notification); a third-party service like OneSignal / Knock (rejected for v1 — the in-app bell is fully achievable on our existing Supabase stack; we can add email/SMS channels later via Resend + the same pipeline); Supabase Realtime Broadcast instead of `postgres_changes` (considered — Broadcast is lower latency but requires per-channel authorization and loses the "source of truth in the table" property; keeping `postgres_changes` so the database table is always authoritative).
**Reasoning.** Keeping the database table as the single source of truth means: (a) the bell never needs to care whether the app was running when the event fired — on next mount it just reads the latest row, (b) role-targeted `type` + CHECK + guard trigger means no future code path can emit a student notification type to an admin bell (or vice-versa), (c) using the existing hooks means migration is incremental and low-risk, (d) `postgres_changes` filtered by `user_id` scales to thousands of concurrent users per institution with Supabase's current limits.
**Regression clauses respected.** 3.9, 3.17, 3.29 (existing notifications rows + existing hooks + existing realtime wiring all preserved; `notification_preferences` table untouched; change is purely additive).

### ADR-21 — RTL mirroring for the full-width header + primary-nav

**Decision.** The `GlobalHeader` uses Flexbox with logical properties exclusively — `ms-6` / `me-6` for horizontal gaps, `ps-*` / `pe-*` for padding — so every layout flip is automatic when `<html dir="rtl">` is set by `LanguageSwitcher`. The notification bell's unread badge uses `end-0 top-0` positioning so the red dot always sits on the outward side regardless of locale. The primary-nav row uses native flex + `gap-2` so items reverse order naturally under RTL (no `flex-row-reverse` hack needed). Text alignment uses `text-start` / `text-end` (never `text-left` / `text-right`). Icons inside the right cluster (search, bell, chevron) use their semantic direction so the chevron on the profile dropdown always points "down" regardless of locale (icons are not mirrored). Date / number formatting inside the bell's popover uses `src/lib/i18nHelpers.ts` with the `ar-QA` locale per ADR-11.
**Alternatives considered.** Per-locale JSX variants (rejected — duplicates markup twice per header); `dir="ltr"` forced on the header regardless of locale (rejected — breaks mirroring expectation documented in clause 2.30); CSS physical overrides (rejected — violates project's logical-property rule in `project-conventions.md`).
**Reasoning.** Logical properties are the W3C-recommended approach for RTL layout; they reduce surface area for RTL-specific bugs to near zero as long as contributors stay disciplined. Locking the header to this rule sets a precedent the rest of the codebase already follows.
**Regression clauses respected.** 3.10, 3.26, 3.30.

### ADR-22 — Single profile entry point; remove the sidebar Profile `NavLink` ahead of the header rollout

**Decision.** Delete the `{ to: "/{role}/settings/profile", icon: UserCircle, label: "Profile" }` item from every role's sidebar nav array (`AdminLayout.tsx`, `CoordinatorLayout.tsx`, `TeacherLayout.tsx`, `StudentLayout.tsx`, `ParentLayout.tsx`) immediately — ahead of the Phase 7 `GlobalHeader` rollout — so the single-entry-point invariant for Profile (parallel to the Settings invariant in ADR-19) holds even during the transition period. The top-bar profile dropdown (clauses 2.13, 2.28) is and remains the only surface that exposes the profile route in the chrome. No route changes; no page component changes; only the duplicate sidebar `NavLink` is removed. When Phase 7 lands and `GlobalHeader` replaces the per-role sidebars, `src/lib/navItems.ts` is defined without any profile entry so the invariant remains automatically.
**Alternatives considered.** Wait for Phase 7 to remove the sidebar Profile item together with the rest of the sidebar (rejected — leaves a visibly duplicated affordance in the chrome for an unknown amount of time while Phase 7 is in flight); keep the sidebar Profile entry and accept the duplication (rejected — user explicitly called it out as unnecessary).
**Reasoning.** A one-line diff (remove one array entry per layout) collapses five duplicate Profile entry points to zero right now, with zero risk of regressing the profile route itself.
**Regression clauses respected.** 3.9, 3.13, 3.31.

### ADR-23 — Admin-dashboard black-area root-cause fix

**Decision.** Harden the page-background contract so the void beneath short Admin dashboard content can never surface. Three concrete changes: (a) every role layout's outer `<main>` is wrapped in a `min-h-screen` container with the design-system page background token (`bg-slate-50` in light / `bg-background` via `.dark` once ADR-01 lands), so the light tone reaches the bottom of the viewport even when dashboard content is short; (b) `src/index.css` adds `body { background-color: hsl(var(--background)); }` so any DOM node without a parent-provided background inherits the design-system tone rather than the OS/browser default; (c) an AST / style-scan property test (piggy-backs on the Task 88 `darkModeTokens.property.test.ts`) verifies no dashboard component uses a hard-coded `bg-black`, `bg-gray-900`, or `bg-slate-900` utility. In addition, the Admin dashboard adopts `<WelcomeHero role="admin" ... />` as its first child (already planned in Task 47) so the page opens with a composed hero that adds vertical weight above the fold and reduces the perceived void below.
**Alternatives considered.** Setting a fixed `min-h-screen` background only on the root `<App />` (rejected — breaks the layout separation where each role layout owns its chrome); injecting a background-only CSS class inline on the Admin dashboard page (rejected — doesn't generalize to the other role dashboards that will hit the same pattern once content density diverges); using `bg-white` instead of `bg-slate-50` on the page (rejected — conflicts with the documented `bg-slate-50` sidebar-layout pattern in the design system).
**Reasoning.** The defect is a page-background-not-reaching-the-fold regression, not a content defect. Fixing it at the layout/CSS contract level means every future role dashboard inherits the fix automatically; fixing it at the component level would paper over the same symptom in five places.
**Regression clauses respected.** 3.1, 3.9, 3.21, 3.32.

## 3.5 Notification type enum (clause 2.29 / ADR-20 detail)

The `notifications.type` CHECK constraint enumerates:

- **Student** (8): `grade_released`, `new_assignment`, `badge_earned`, `streak_at_risk`, `peer_milestone`, `perfect_day_nudge`, `prerequisite_unlocked`, `digest`
- **Teacher** (4): `grading_queue_new`, `course_announcement_required`, `at_risk_student_flagged`, `digest`
- **Coordinator** (4): `cqi_plan_review_due`, `curriculum_gap_flagged`, `outcome_attainment_drop`, `digest`
- **Admin** (5): `pending_approval`, `rate_limit_exceeded`, `system_alert`, `user_onboarded`, `digest`
- **Parent** (5): `child_grade_released`, `child_at_risk`, `child_streak_milestone`, `fee_due`, `digest`

`digest` is shared across all roles (the aggregate weekly summary). The per-role guard trigger consults a map in the trigger body:

```
ROLE_TO_ALLOWED_TYPES = {
  student     : {grade_released, new_assignment, badge_earned, streak_at_risk, peer_milestone, perfect_day_nudge, prerequisite_unlocked, digest},
  teacher     : {grading_queue_new, course_announcement_required, at_risk_student_flagged, digest},
  coordinator : {cqi_plan_review_due, curriculum_gap_flagged, outcome_attainment_drop, digest},
  admin       : {pending_approval, rate_limit_exceeded, system_alert, user_onboarded, digest},
  parent      : {child_grade_released, child_at_risk, child_streak_milestone, fee_due, digest}
}
```

### 5.2 `supabase/migrations/20260901000002_add_handle_new_user_trigger.sql`

```sql
-- Auto-insert a matching profiles row when a new auth.users row is created.
-- SECURITY DEFINER so the function bypasses RLS; the function itself is the only
-- path that can insert a profiles row from a freshly-signed-up user.

CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS trigger
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
BEGIN
  INSERT INTO public.profiles (
    id,
    email,
    full_name,
    role,
    institution_id,
    theme_preference,
    tour_completed_at,
    created_at,
    updated_at
  )
  VALUES (
    NEW.id,
    NEW.email,
    COALESCE(NEW.raw_user_meta_data ->> 'full_name', ''),
    'student',                                                  -- default role, can be changed by admin
    NULLIF(NEW.raw_user_meta_data ->> 'institution_id', '')::uuid,
    'system',
    NULL,
    now(),
    now()
  )
  ON CONFLICT (id) DO NOTHING;

  RETURN NEW;
END;
$$;

REVOKE ALL ON FUNCTION public.handle_new_user() FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.handle_new_user() TO postgres, service_role;

DROP TRIGGER IF EXISTS on_auth_user_created ON auth.users;
CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW
  EXECUTE FUNCTION public.handle_new_user();
```

### 5.3 `supabase/migrations/20260901000003_tighten_avatars_bucket_policies.sql`

```sql
-- The avatars bucket already exists (20260416193532_create_storage_buckets.sql).
-- Tighten INSERT/UPDATE policies to the user's own folder and add a symmetric DELETE policy.

DROP POLICY IF EXISTS "avatars_auth_upload" ON storage.objects;
DROP POLICY IF EXISTS "avatars_auth_update" ON storage.objects;
DROP POLICY IF EXISTS "avatars_auth_delete" ON storage.objects;

CREATE POLICY "avatars_auth_upload" ON storage.objects
  FOR INSERT TO authenticated
  WITH CHECK (
    bucket_id = 'avatars'
    AND (storage.foldername(name))[1] = (select auth.uid())::text
  );

CREATE POLICY "avatars_auth_update" ON storage.objects
  FOR UPDATE TO authenticated
  USING (
    bucket_id = 'avatars'
    AND (storage.foldername(name))[1] = (select auth.uid())::text
  )
  WITH CHECK (
    bucket_id = 'avatars'
    AND (storage.foldername(name))[1] = (select auth.uid())::text
  );

CREATE POLICY "avatars_auth_delete" ON storage.objects
  FOR DELETE TO authenticated
  USING (
    bucket_id = 'avatars'
    AND (storage.foldername(name))[1] = (select auth.uid())::text
  );

-- SELECT remains as defined by 20260504033018_restrict_avatars_bucket_listing.sql
-- (authenticated can SELECT only their own folder; public serving of avatars works

    ARRAY[]::uuid[],

    now(),
    now()

       AND onboarding_progress.self_efficacy_completed = false

       AND onboarding_progress.study_strategy_completed = false

The `ON CONFLICT` behavior of every INSERT is either `DO NOTHING` (append-only) or `DO UPDATE` with guards like the onboarding-progress guard above so re-running never overwrites authenticated user changes.

## 7. Rollout plan

Safe order, merge-by-merge, each phase merge-safe and backward-compatible:

1. **Phase 1 — Migrations + scale-realistic seed.** Merge `20260901000001` (tour + theme_preference), `20260901000002` (handle_new_user trigger), `20260901000003` (avatars bucket policies), `20260901000004` (scale-realistic seed: 3 institutions × 500+ students each), `20260901000005` (invitations table), `20260901000006` (institution join modes), `20260901000007` (language_preference), and `20260901000008` (rate-limiting audit). Each is additive or idempotent; `handle_new_user()` only fires on future `auth.users` inserts (no effect on existing rows). Regen types (`pwsh scripts/regen-types.ps1`) and commit `src/types/database.ts`. Deploy to Supabase.
2. **Phase 2 — Shared components + auth surfaces behind feature flags where needed.** Land `WelcomeHero`, `TopBar`, `ProfileDropdown`, extended `EmptyState`, `GuidedTour`, `MondayWeekPicker`, `AvatarUpload`, `ThemeToggle`, the new stores (`themeStore`, `tourStore`), `mapSupabaseError`, `useStandardMutation`, `useOptimisticToggle`, `useTheme`, `useAvatarUpload`, `useGuidedTour`, `useInstitutionBrowse`, `useInviteUsers`, plus the `/signup` route, the `/accept-invite/:token` route, and the Admin Invite Users page and its `send-invitation-email` Edge Function. Ship with tests. The guided tour is wired but only auto-launches when `profiles.tour_completed_at IS NULL` and the user opts in — during phases 2–4 the Zustand `tourFeatureFlag` keeps auto-launch off globally so phases 3–4 can land role dashboards without competing for the user's attention.
3. **Phase 3 — Role dashboards one at a time: Admin → Coordinator → Teacher → Parent → Student.** The Student dashboard is last because it also carries the onboarding-completion / performance fixes (clause 2.20) and touches the existing hero preserved by clause 3.21. Each role's PR replaces the bespoke top bar with `<TopBar />`, adds `<WelcomeHero />` as the first child of the dashboard body, adopts `<EmptyState />` on every list surface, converts any backend-bound Switch to `useOptimisticToggle`, and wires the per-role guided tour.
4. **Phase 4 — i18n coverage sweep, gated by `i18nCoverage.property.test.ts`.** Run `i18next-cli` extraction, reconcile Arabic translations per `src/locales/ar/glossary.md` (MOEHE MSA glossary), and land the property test. Once the test is green, the gate stays in CI forever. The translation memory file at `src/locales/_translation-memory.json` (ADR-17) is committed so future refactors don't lose existing translations.
5. **Phase 5 — Enable `tourFeatureFlag`, verify across all 5 roles, then delete `QuickStartChecklist.tsx` and `WelcomeTour.tsx`.** Direct deletion — no compliance or change-management reason to keep both production-side once the tour is green on all 5 layouts and `authSurfaces.property.test.ts` has passed for one release.
6. **Phase 6 — Post-launch monitoring.** Watch `audit_logs` for `signup_rate_limit_exceeded` and `invite_accept_rate_limit_exceeded` events; watch Vercel Analytics for onboarding-complete navigation p95 < 200 ms and Student dashboard TTI p95 < 1500 ms at 500-user and 5000-user institution scale. Tune rate-limit thresholds (ADR-16) and any query `staleTime` values based on observed traffic.

### Kill switches

- **Theme.** `profiles.theme_preference` still accepts `'system'` — users stuck on broken dark mode can revert to system-follows-OS with one click in the profile dropdown; `themeStore` always honors the DB value over the local Zustand value on app boot.
- **Tour.** The `useGuidedTour().skip()` action writes `profiles.tour_completed_at = now()` so any user can opt out immediately; the tour can be re-launched from the profile dropdown (`Take the tour`).
- **Demo seed.** Re-runnable any time without side-effects because every INSERT uses `ON CONFLICT DO NOTHING` (or a guarded `DO UPDATE` for `onboarding_progress`).
- **Onboarding optimistic nav.** If the server-side `process_onboarding` fails, the non-blocking Retry action in the Sonner toast re-invokes the Edge Function without forcing the user back into the wizard (clause 2.20). The wizard itself remains mountable as a fallback.
- **Avatar upload.** Files failing type/size validation never reach the server (Zod on the client), and the server-side bucket MIME + `file_size_limit = 2097152` (2 MB) remain as a second line of defense.

## 8. Resolved decisions

Scalability-first answers to every previously-open question. Each decision is already reflected in the ADRs (§3), file plan (§4), migrations (§5), and rollout plan (§7). Listed here as the traceable record of how the design resolves each question.

### 8.1 Default signup role → `student`, with invite-token system for non-student onboarding

Self-service signup at `/signup` creates `profiles.role = 'student'` by default (public tier). Teachers, Coordinators, Admins, and Parents onboard via the invite-token system specified in ADR-12:

- New `invitations` table keyed by `(institution_id, email, role)` with a signed token, 7-day expiry, and single-use enforcement.
- `/accept-invite/{token}` route validates the token server-side, reads the target role and institution, and creates the profile with the invited role instead of the default.
- Admin "Invite Users" page (`src/pages/admin/users/InviteUsersPage.tsx`) lets admins bulk-invite by CSV with role + institution.
- Rate-limiting (ADR-16) prevents abuse: 5 signup attempts / IP / hour via Supabase Auth's built-in `GOTRUE_RATE_LIMIT_EMAIL_SENT`, plus CAPTCHA challenge on threshold approach.
- Migration `20260901000005_create_invitations_table.sql` defines the table + RLS (admin-insert scoped to `institution_id = auth_institution_id()`, public-select-by-token via SECURITY DEFINER RPC), and `supabase/functions/send-invitation-email/index.ts` dispatches the email through Resend.

### 8.2 Institution selection → multi-institution self-serve with per-institution join modes

ADR-13 formalizes multi-institution self-serve. `institutions` gains `slug text UNIQUE NOT NULL`, `logo_url text`, `allowed_email_domains text[]`, and `join_mode text CHECK (… IN ('open','invite_only','domain_restricted')) DEFAULT 'invite_only'`.

- `/signup` is a two-step flow: step 1 choose-institution (searchable Shadcn `Command` list populated from `institutions WHERE is_active = true` via `useInstitutionBrowse`), step 2 account form.
- `open` mode allows self-signup without an invite but sets `profiles.status = 'pending_verification'` until the email is verified (ADR-14).
- `invite_only` mode rejects self-signup with "This institution requires an invitation — ask your admin".
- `domain_restricted` mode auto-validates the email domain against `institutions.allowed_email_domains`.
- `handle_new_user()` trigger (extended in migration `20260901000002`) reads `institution_id` + `role` from `raw_user_meta_data`, validates against the institution's `join_mode`, writes `profiles.status = 'active' | 'pending_verification'`, and inserts a `pending_approval` audit-log row for admins when the role is not `student`.
- Migration `20260901000006_add_institution_join_modes.sql` adds the three new columns with `join_mode = 'invite_only'` as the default so existing institutions do not silently open to self-signup.

### 8.3 Dark mode default for unauthenticated users → `system` (follow OS via `prefers-color-scheme`)

`themeStore` on mount (before auth) reads `localStorage.theme` or falls back to `window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light'`. After auth, `profiles.theme_preference` replaces `localStorage` as the source of truth. On sign-out, the `localStorage` value is retained so the next sign-in page matches what the user just saw (no flash). This resolves to the `'system'` default already wired into migration `20260901000001`.

### 8.4 Tour language → follow `i18n.language`, with opt-in override via `profiles.language_preference`

Per-role step arrays in `src/lib/tours/{role}Tour.ts` use `t()` for every `title` and `content` field. `react-joyride` re-keys steps when `i18n.language` changes, so switching locale mid-tour re-renders the current step in the new language.

`profiles.language_preference text CHECK (… IN ('en','ar','auto')) DEFAULT 'auto'` (migration `20260901000007`) lets a user pin the tour to a specific language regardless of the session's `i18n.language`. `'auto'` falls back to `i18n.language`. The Profile page exposes this preference alongside theme preference.

### 8.5 Parent tour copy — 7-step array, with all 5 role tours normalized to 7 steps for parity

For parity across roles, every role tour is standardized at 7 steps. The Parent tour:

1. `parent.tour.linkedStudents` — "Your linked students appear here"
2. `parent.tour.academicProgress` — "Track their grades and attainment"
3. `parent.tour.habitTracker` — "See how they're building study habits"
4. `parent.tour.feeOverview` — "Manage school fees and payments"
5. `parent.tour.notifications` — "Stay informed about important updates"
6. `parent.tour.planner` — "View upcoming assignments and events"
7. `parent.tour.settings` — "Adjust notification preferences and language"

The Admin, Coordinator, Teacher, and Student tours are each extended to 7 steps (previous design implicitly assumed 5–6). Each `src/lib/tours/{role}Tour.ts` file header lists the 7 step-target `data-tour` anchors that layouts and dashboards must render.

### 8.6 `DataTable` header contrast under `.dark` → `bg-muted text-foreground` (tokens, both themes)

`src/components/shared/DataTable.tsx` sets `bg-muted dark:bg-muted text-foreground dark:text-foreground` on the header row. The `darkModeTokens.property.test.ts` oracle asserts this exact mapping: header-row computed background equals the resolved `--muted` value and header-row computed text color equals the resolved `--foreground` value under both `:root` and `.dark`.

### 8.7 Quick Start widget removal → direct deletion in phase 5 (no feature-flag holdback)

Confirmed deletion in phase 5. Feature-flag gating (Zustand `tourFeatureFlag`) applies during phases 2–4 only. There is no compliance or change-management reason to keep both the widget and the tour in production simultaneously after phase 5 verification.

### 8.8 Demo seed → scale-realistic (3 institutions × 5 programs × 20 courses × 500 students each)

`20260901000004_seed_demo_data.sql` is revised to scale-realistic scope:

- **3 demo institutions**: Qatar University Demo, Carnegie Mellon Qatar Demo, Texas A&M Qatar Demo.
- **Per institution**: 5 programs, 4 courses per program (= 20 courses per institution), 500 students, 50 teachers, 5 coordinators, 2 admins, 200 parents.
- **Outcome chains**: realistic ILO/PLO/CLO chains with sensible `outcome_mappings` (weights summing to 100 per parent).
- **History depth**: 6 months of `xp_transactions`, `evidence`, `habit_tracking`, grades, and notifications for the demo student subset.
- **Platform-wide state**: badge spotlight history, leaderboard state, past surveys, and audit-log entries reflecting realistic operation.
- **Deterministic UUIDs**: derived from stable namespaces so re-running the seed is idempotent and safe.
- **Onboarding state**: exactly one demo student is left at `onboarding_progress.current_step = 'welcome'` so clause 2.14 has a concrete fixture; all other students are randomly-progressed through onboarding so the admin Pending Onboarding page has realistic data to render (and the Remind button can be exercised against real rows).

This supports both the clause-2.14 first-time-login scenario and the ADR-13 multi-institution scalability story in a single idempotent seed.
