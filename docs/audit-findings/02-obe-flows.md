## Confirmed Issues (ranked)

### BLOCKER ‚Äî none that hard-crash the core grade‚Üíattainment loop

The live attainment loop (grade ‚Üí trigger ‚Üí `outcome_attainment`) is intact via the DB trigger, so there is no blocker for the primary teaching flow. The blocker-class problems are confined to the **accreditation/course-file exports** (below), which are core to an OBE platform's value proposition.

### HIGH

**H-1 ‚Äî `outcome_mappings` weight "sum-to-100" invariant is not enforced anywhere, and the scale is inconsistent (0‚Äì1 vs 0‚Äì100).**
Domain rule: "All PLO‚ÜíILO weights for a given PLO must sum to 100" (`domain-knowledge.md`).

- Zod schemas use a **0‚Äì1** scale and never check the sum: `src/lib/schemas/plo.ts:14-18`, `src/lib/schemas/clo.ts:18-22` (`weight: z.number().min(0).max(1)`).
- UI only emits a **soft toast warning** at `>1.0` / `<0.5`, and still saves: `src/pages/coordinator/plos/PLOForm.tsx:397-401, 425-447`; `src/pages/teacher/clos/CLOForm.tsx:607-610`.
- No DB CHECK/trigger on `outcome_mappings` enforces a per-child sum (grep across `supabase/migrations` for `outcome_mappings` shows indexes/cascade reads only).
- Meanwhile the attainment cascade and the property test fixtures assume **0‚Äì100** (`‚Ä¶20260520070203_‚Ä¶sql` uses `om.weight` as-is; `src/__tests__/properties/_generators/outcomes.ts:13-19` "sum to exactly 100").
  **Impact:** weighted PLO/ILO rollups are mathematically arbitrary (weights need not sum to a fixed base), and the 0‚Äì1 UI scale silently diverges from the 0‚Äì100 used elsewhere. Property tests pass against generators, not against product validation.
  **Fix direction:** pick one scale (recommend 0‚Äì100 to match docs), add a shared zod `superRefine` sum check, and a DB trigger/constraint on `outcome_mappings` per `(target_outcome_id)` group.

**H-2 ‚Äî Attainment **scope mismatch** between writer and aggregate readers (admin heatmap, CLO headline).**
Writer (trigger) stores: CLO ‚Üí `scope='student_course'`, PLO ‚Üí `scope='course'`, ILO ‚Üí `scope='program'` (`‚Ä¶20260520070203_rewrite_attainment_trigger_pure_sql.sql`, INSERT scopes).
Readers expect different scopes:

- `useAdminPLOHeatmap.ts:113` reads PLO at `scope='program'` (PRIMARY path) ‚Äî but PLOs are written at `'course'` ‚áí PRIMARY never matches.
- `useAdminPLOHeatmap.ts:161` fallback reads CLO at `scope='course'` ‚Äî but CLOs are written at `'student_course'` ‚áí fallback finds nothing ‚áí PLOs render as **unmeasured grey (`-1`)**.
- `useCLOs.ts:500` `useCLOAttainment` reads CLO at `scope='course'` ‚áí CLO detail headline likely always "no data".
  **Impact:** despite 1,743 attainment rows, admin PLO heatmap and CLO-detail headlines probably show no data, while student dashboards (which read `student_course`) work. **Flagged for live verification** ‚Äî confirm the scope distribution of `outcome_attainment` before remediation (I could not query the DB).
  **Fix direction:** standardize the scope taxonomy and align readers; or have the trigger also write course-scope CLO and program-scope PLO aggregates.

**H-3 ‚Äî Accreditation report attainment is always zero (schema drift on `outcome_attainment`).**
`supabase/functions/generate-accreditation-report/index.ts`:

- Filters `a.scope === 'PLO'` / `'ILO'` (`:574-576`, `:592-594`). The enum is `student_course|course|program|institution` (`database.ts:7350`, `‚Ä¶create_evidence_attainment_tables.sql:20`) ‚Äî these strings never occur ‚áí `records` always empty ‚áí `avgAttainment=0`, `evidenceCount=0`.
- Selects/uses `score_percent` (`:570-585`) but the column is `attainment_percent` (`database.ts:3328`).
  **Impact:** ABET/HEC/QQA/etc. PDFs render but every PLO/ILO row is 0% ‚Äî accreditation output is misleading/unusable.

**H-4 ‚Äî Course file generator references multiple non-existent columns (will error or emit empty sections).**
`supabase/functions/generate-course-file/index.ts`:

- `outcome_mappings` selected as `child_outcome_id, parent_outcome_id` (`~:470`) ‚Äî real columns are `source_outcome_id, target_outcome_id` (`database.ts:3381+`) ‚áí CLO‚ÄìPLO mapping section empty/errors.
- `outcome_attainment ‚Ä¶ .eq('scope','CLO')` (`~:560`) ‚Äî invalid scope ‚áí CLO attainment empty.
- `assignments ‚Ä¶ select('‚Ä¶ clo_ids')` (`~:520`) ‚Äî assignments expose `clo_weights` (`database.ts:250`), not `clo_ids` ‚áí linked-CLO column blank.
- `cqi_action_plans ‚Ä¶ select('title, gap_description, corrective_actions, status')` (`~:600`) ‚Äî none of `title/gap_description/corrective_actions` exist (real: `action_description`, `root_cause`, `responsible_person`, ‚Ä¶, `database.ts:1550-1566`) ‚áí **PostgREST error ‚Üí function 500**.
  **Impact:** the per-course accreditation file (a headline OBE deliverable) is effectively broken.

### MEDIUM

**M-1 ‚Äî Three independent implementations of the same rollup (maintenance hazard / drift risk).**
(1) Active DB trigger `trigger_attainment_rollup()` (pure SQL); (2) dormant edge function `calculate-attainment-rollup`; (3) client `src/lib/quizEvidence.ts`. They classify identically (85/70/50) but differ in cascade depth (quiz path = CLO only) and in GA handling. Only the trigger runs in production today.
**Fix direction:** designate one source of truth (the trigger), delete/retire the others or have them call the trigger logic.

**M-2 ‚Äî GA cascade in edge/report code uses drifted `ilo_id` column.**
`calculate-attainment-rollup/index.ts` (GA step 6) and `generate-accreditation-report/index.ts:690-705` query `graduate_attribute_mappings.ilo_id`. Generated types only have `outcome_id` (`database.ts:2210-2231`); `ilo_id` exists only as a backfilled drift column (`‚Ä¶20260526145124_fix_graduate_attributes_schema_drift.sql`). Hooks correctly use `outcome_id`. Latent because mappings=0 today, but will silently mis-resolve once GAs are mapped.

**M-3 ‚Äî Curriculum matrix coverage is a placeholder, not attainment-driven.**
`useCurriculumMatrix.ts:24-28` marks a cell green purely on CLO-mapping count; the comment states it's a placeholder "until attainment data is available." Coordinators cannot see _attainment_ gaps from the matrix, only mapping presence.

### LOW

**L-1 ‚Äî `useUpdateCLO` bypasses generated types with `supabase as any`** (`useCLOs.ts:182`, eslint-disabled at `:181`) pending a types regen for `tutor_autonomy_level`. Works, but defeats type-safety on that write path.

**L-2 ‚Äî `generate-course-file` self-creates the `reports` bucket on upload failure** (`index.ts` upload retry) whereas the accreditation report treats a missing bucket as a hard error (`generate-accreditation-report/index.ts` upload block). Inconsistent storage assumptions across two sibling functions.

**L-3 ‚Äî `useAdminPLOHeatmap` documented PRIMARY ("program") derivation is effectively dead** under the current trigger scopes (see H-2); the code path exists but never fires for trigger-produced data. Cleanup once H-2 is resolved.

## Improvements (works, but could be better)

- **Outcome management hooks are clean and consistent** ‚Äî explicit column selects, batched joins (no N+1), audit logging on every mutation, dependency guards on delete (`useILOs.ts:150-171`, `usePLOs.ts:175-205`). Good baseline to build the weight-sum enforcement onto.
- **`attainmentClassifier` is the single, correct threshold authority** and is reused by the heatmap, GA manager, and outcome chain. Consider routing the edge functions and the DB trigger through the _same_ constants (currently each re-hardcodes 85/70/50) to prevent future drift.
- **`useOutcomeChain` / `outcomeChain.ts`** is a model implementation: pure assembler split from I/O (unit-testable), batched queries, correct mapping direction, unified empty state. Use it as the reference pattern.
- **`useAdminPLOHeatmap` derivation is well-documented** (PRIMARY ‚Üí CLO-rollup ‚Üí none with `-1` sentinel). Once the scope mismatch (H-2) is fixed, this becomes a strong feature; consider surfacing `derivation` in the UI so admins know whether a number is program-level or rolled up.
- **CQI manager** already models the full closing-the-loop lifecycle including the newly added `root_cause`/`due_date`/`evidence_of_improvement`. Adding a baseline‚Üíresult delta indicator (already partially present, `CQIManager.tsx:~470`) to the coordinator dashboard would close the analytics gap.
- **Rubrics** list resolves CLO titles in one round trip and paginates ‚Äî extend the same join pattern to the course-file CLO/PLO mapping section after fixing H-4.
- **Edge functions need a CI schema-contract check.** H-3/H-4/M-2 are all "compiles in Deno, wrong column at runtime" ‚Äî a generated-types lint or a smoke test that actually invokes each function against a seeded DB would have caught every one.
