## Confirmed Issues (ranked)

### BLOCKER

**B-1 ‚Äî The entire individual-badge subsystem is non-functional: badges are written to `student_badges`, the UI reads `badges`, and `award-xp` never triggers badge checks.**

Three compounding defects:

1. **Table divergence (write vs read).** `check-badges` inserts each earned individual badge into **`student_badges`** with a _text_ `badge_id` key (e.g. `"streak_7"`):

   - write: `supabase/functions/check-badges/index.ts:1870-1885` (`from("student_badges").insert({ student_id, badge_id, awarded_at })`)
   - existing-set read: `check-badges/index.ts:1709-1714` (`from("student_badges").select("badge_id")`)
   - schema: `student_badges.badge_id text` `20260601205749_deploy_phantom_tables_function_buckets_rls.sql:108-115`.
     The student UI reads a **different** table, `badges`: `src/hooks/useBadges.ts:22-24` (`from("badges").select("badge_key, badge_name, emoji, awarded_at")`). `BadgeCollection.tsx` renders from `useStudentBadges`. **Result: any badge `check-badges` awards is invisible to the student.** (The only place `check-badges` writes the read-by-UI `badges` table is the _tier_ upgrade path `check-badges/index.ts:1562-1575` and _team_ badges `awardTeamBadges` ‚Äî not the ~25 individual badges.)
   - Live corroboration: `student_badges=0` AND `badges=0` ‚Äî no individual badges have been persisted to either table for 71 students with 4117 XP txns.

2. **`award-xp` does not fan out to `check-badges`.** Grep for `invoke("check-badges"` across `supabase/functions/award-xp/**` returns **0 matches**. The domain rule "Badges are checked idempotently after XP award, submission, or streak update" is therefore unmet for the highest-volume paths. The only callers of `check-badges` are 5 niche flows: `process-streak` (team + comeback), `process-onboarding`, `usePlannerTasks.ts:227`, `useSessionCompletion.ts:161`, `useWeeklyGoalXP.ts:144`, `useWellnessHabits.ts:83`. **No caller fires for login, on-time submission, grade release, or streak milestones** (the core academic loop). `submissions`/`grades` hooks invoke neither `award-xp` nor `check-badges` (grep: 0 hits in `useSubmissions.ts`); grade XP comes only from the DB trigger (which also doesn't check badges).

3. **Net effect.** Even where `check-badges` _is_ invoked, the badges it awards land in `student_badges` and never surface in the UI. The badge feature is effectively dead end-to-end. This is the single most material gamification defect and explains `badges=0`/`student_badges=0` live.

**Fix direction:** (a) pick ONE badges table ‚Äî given the UI and 3 live writers (`badges`), standardize `check-badges` individual-badge inserts onto `badges` (with `badge_key`/`badge_name`/`emoji`), or add a view/migration; (b) have `award-xp` invoke `check-badges` (fire-and-forget) after a successful award with an appropriate trigger, and wire submission/grade flows to `check-badges`.

### HIGH

**H-1 ‚Äî `xp_total` is maintained by two independent mechanisms that can silently diverge, and the DB grade-trigger path never recomputes `level`.**
Domain invariant: "xp_total is derived from SUM(xp_transactions)".

- The **edge function** recomputes the true SUM on every award and upserts it: `award-xp/index.ts:1113-1166` (Step 7 `select xp_amount ‚Ä¶ reduce`; Step 9 upsert `xp_total`, `level`).
- The **DB trigger** `trigger_attainment_rollup()` (the _live_ writer on grade release) instead does a blind increment and **omits level**: `20260520094217_fix_trigger_xp_idempotent.sql:105-113` ‚Äî `INSERT ‚Ä¶ 'grade' ‚Ä¶ ON CONFLICT DO NOTHING; ‚Ä¶ UPDATE student_gamification SET xp_total = xp_total + 15`. `check-bonus-question/index.ts:170-185` is a third writer (recomputes SUM but applies no level/multipliers).
- There is **no DB trigger** that derives `xp_total` from `SUM(xp_transactions)` (grep `CREATE TRIGGER ‚Ä¶ xp_transactions` shows only the append-only immutability guard `20260520102905`/`20260520103007`, not a sum-derivation).
  **Impact:** the grade trigger adds a flat 15 (ignores adaptive multipliers the edge fn would apply, and ignores level-ups), so `xp_total` and `level` drift from the canonical SUM whenever grades drive XP. With 859 grades live (per the OBE audit), this is a real, ongoing source of drift. The invariant holds only on the edge-function path.
  **Fix direction:** make the SUM the single source of truth ‚Äî either route grade XP through `award-xp`, or replace the increment with a recompute + `calculateLevel` in the trigger; add a reconciliation/CHECK to assert `xp_total = SUM`.

**H-2 ‚Äî The daily streak driver (`process-streak`) is never correctly invoked, so streaks don't increment/reset day-to-day.**

- Login only writes an activity row: `src/providers/AuthProvider.tsx:255-259` (`logActivity({event_type:"login"})`). There is **no** client `process-streak` invoke and **no** DB trigger on `student_activity_log` to award login XP or advance the streak (grep: none).
- The remaining driver is the Vercel cron `api/cron/streak-reset.ts:8` which calls `invokeEdgeFunction('process-streak', { type: 'midnight_reset' })` ‚Äî **with no `student_id`**. But `process-streak.validatePayload` requires `student_id` and returns 400 otherwise: `process-streak/index.ts:155-167`. So the cron call fails validation and processes **no** students. (The function has no "all students" batch branch ‚Äî it is single-student only.)
- The old pg_cron HTTP job `streak-midnight-reset` was intentionally unscheduled (migration-history-reconciliation Task 15), leaving Vercel as the sole ‚Äî and broken ‚Äî driver.
  **Impact:** `streak_current` is not advanced on login and not reset on misses through any working path; streak milestones (7/14/30/60/100 ‚Üí XP + badge checks) effectively never fire automatically. Streak values shown on the dashboard come from whatever `process-streak` ran historically. **Flagged for live verification** (confirm whether any other scheduler invokes per-student `process-streak`).
  **Fix direction:** either (a) call `process-streak` per-student on login (client or a login DB trigger), or (b) add an "all active students" batch mode to `process-streak` and have the midnight cron iterate.

**H-3 ‚Äî `badge_definitions` is created but never seeded (and never read by the award path), so DB-driven badge management and the weekly Badge Spotlight are inert.**

- Table + RLS created: `20260414043939_‚Ä¶add_badge_definitions_columns.sql:11-47`. **No seed exists** ‚Äî grep for `INSERT INTO badge_definitions` / `from("badge_definitions")` returns **0** across migrations, `supabase/seed.sql`, and `scripts/`. (`seed.sql` seeds the `badges` table and `badge_spotlight_schedule`, never `badge_definitions`.)
- The award path does **not** depend on it: `check-badges` uses an in-file hard-coded `BADGE_XP` map (`check-badges/index.ts:38-78`) and string keys, so the **NO-DATA on `badge_definitions` is NOT what blocks badge awards** (that's B-1).
- The real consumer is the spotlight rotation: `get_badge_spotlight(‚Ä¶)` does `FROM badge_definitions WHERE is_archived = false ‚Ä¶` (`20260414043956_‚Ä¶create_get_badge_spotlight_function.sql:22-26`) and `badge_spotlight_auto_rotate()` (pg*cron `0 0 * * 1`, `20260720000007*‚Ä¶:12`). With 0 rows the eligible set is empty ‚áí spotlight always null ‚áí the 2√ó spotlight multiplier in `award-xp/index.ts:951-1000`can never trigger.
**True severity:** Medium-High. It does **not** block earning/among badges (B-1 already does that), but it makes the Badge Spotlight feature and any admin "DB-driven badge definitions" UI permanently empty. Seeding`badge_definitions`is necessary for spotlight, and would be required if the badge system were ever migrated to be definition-driven.
**Fix direction:** add a per-institution seed migration for`badge_definitions`mirroring`src/lib/badgeDefinitions.ts`, then reconcile the award path to read it (ties into B-1's "one source of truth").

**H-4 ‚Äî Team gamification is split across two tables/engines (`teams` columns vs `team_gamification`), and the app explicitly distrusts `team_gamification`.**

- Edge functions read/write **`team_gamification`**: `process-streak/index.ts:236-330` (team streak upsert), `check-badges` team block (`team_gamification.xp_total/streak_current`), `team-streak-risk-cron/index.ts:28-31`. Table exists in a phantom-tables migration: `20260601205749_‚Ä¶:73-100`.
- The app hook **`useTeams`** reads team XP/streak from the **`teams`** table and states the other table is absent: `src/hooks/useTeams.ts` `useTeamGamification` comment ‚Äî _"The previously-referenced `team_gamification` table does not exist in production ‚Äî XP and streak fields live on `teams` itself"_ (reads `teams.xp_total, streak_count`).
- A **separate** SQL cron advances `teams.streak_count` directly: `20260415071802_add_team_streak_cron_job.sql:13-34`, awarding `team_badges` (a _third_ badge table). Meanwhile `award-xp` increments `teams.xp_total` via `increment_team_xp` RPC (`20260720000012_‚Ä¶`).
  **Impact:** two divergent team-streak engines on two tables, plus a team-badge table (`team_badges`) distinct from `badges`/`student_badges`. With teams=0 this is inert today, but it is a correctness/consistency landmine the moment teams are used. **Flagged for live verification** (confirm whether `team_gamification` actually exists on live).
  **Fix direction:** consolidate onto one team-gamification source (recommend `teams` since the app trusts it) and retire/replace the `team_gamification` writers.

**H-5 ‚Äî No `perfect_day` (50 XP) award is implemented; only the 6 PM nudge exists.**
Domain rule: "Perfect Day = all 4 completed before midnight ‚Üí 50 XP (`perfect_day`)".

- `perfect_day` is a valid XP source (`award-xp/index.ts:17,160,179`) and a milestone source, but **no code path awards it**: grep for `source: "perfect_day"` / `is_perfect_day` write paths finds only the seed (`supabase/seed.sql:644`), the `habit_tracking.is_perfect_day` column, and label/enum references ‚Äî **no `awardXP({source:"perfect_day"})` call anywhere** in `src/` or `supabase/functions/`.
- The nudge cron computes "almost perfect" and notifies, but explicitly says the reward is elsewhere: `perfect-day-prompt/index.ts:208-220` ("Complete your X to earn 50 bonus XP") ‚Äî yet nothing grants it on the 4th habit.
  **Impact:** students completing all 4 daily habits never receive the 50 XP Perfect Day bonus; the perfect-day economy is advertised (nudge, heatmap star) but not paid out.
  **Fix direction:** on the habit-completion path (when the 4th habit logs), award `perfect_day` (50 XP, idempotent `perfect_day:{student}:{date}`) and fire `check-badges(trigger:"habit_log")`.

### MEDIUM

**M-1 ‚Äî Streak-freeze inventory cap is 3 (DB/RPC), but the domain rule is max 2.**
`process_marketplace_purchase` blocks at `>= 3` (`20260601004718_‚Ä¶sql:58-63`), `process-purchase/index.ts:222-260` re-checks `< 3`, and `useStreakFreeze.ts:95-98` throws at `>= 3`. Domain doc: "Streak Freeze ‚Ä¶ max 2 in inventory." The 200-XP cost is also not hard-enforced as 200 ‚Äî it's whatever the `streak_shield` marketplace item is priced at (the legacy fallback hard-codes `-200`: `useStreakFreeze.ts:79-87`). **Fix:** align the cap to 2 (or update the doc) and assert the 200-XP price on the seeded item.

**M-2 ‚Äî Challenge/team hooks still use `as never` casts on `social_challenges`/`teams`/`challenge_participants`, defeating generated types.**
`useChallenges.ts:49,73,168,‚Ä¶` and several `useTeams.ts`/legacy paths cast `from("social_challenges" as never)` / `as never` inserts. This silences the generated `Insert`/`Row` types and can hide column drift (the same class of bug found elsewhere in this repo). The freshly-fixed create paths (`pickColumns` + `*_INSERT_COLUMNS`) are correct; the legacy/list paths are not. **Fix:** drop `as never` now that the tables exist in generated types.

**M-3 ‚Äî Level-threshold and XP-derivation logic is duplicated in 3+ places (client lib, `award-xp`, DB trigger, `check-bonus-question`).**
`generateLevelThresholds`/`calculateLevel` exist in both `src/lib/xpLevelCalculator.ts:15-62` and `award-xp/index.ts:84-117` (Deno can't import the lib). SUM-derivation is re-implemented in `award-xp` (Step 7), `check-bonus-question` (`:170-185`), and `checkLeaguePromotion` (`award-xp:~360`). Divergence risk if the formula or derivation changes. **Fix:** centralize where feasible; add a property test asserting parity between the client and edge formula tables.

**M-4 ‚Äî XP "spend" (marketplace) is tracked only in `xp_purchases`, not as negative `xp_transactions`, so `student_gamification.xp_total` ‚âÝ spendable balance.**
Balance is computed as `SUM(xp_transactions) ‚àí SUM(xp_purchases)` inside the RPC (`20260601004718_‚Ä¶sql:21-30`), but `xp_total` (shown as the headline number and used for leaderboard/levels) only ever **grows** from `xp_transactions`. This is internally consistent (levels are meant to reflect lifetime XP, balance reflects spendable), but the dual meaning of "XP" is a UX/trust hazard and the marketplace `streak_shield` fallback writes a _negative_ `xp_transactions` row (`useStreakFreeze.ts:79-87`) which **would** reduce `xp_total`/level ‚Äî an inconsistency between the two purchase paths. **Fix:** standardize one debit model; never mix negative `xp_transactions` with the `xp_purchases` ledger.
