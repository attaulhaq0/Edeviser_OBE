0 · How to read this document

This audit was produced in five passes plus a verification pass. Pass 1 traced the React/TypeScript source and the Supabase edge functions for every role, citing file:line. Pass 2 re-ran the material claims against the live production database (schemas, row counts, RLS, XP ledger). Pass 3 (§2.5) authenticated as a real demo user in each role and exercised RLS-scoped reads, the leaderboard RPC, real write paths, and a cross-profile/cross-institution isolation matrix. Pass 4 (§2.6) went further: for every empty feature domain it seeded live data through the real flow, invoked the dependent edge function, and observed the actual result. Pass 5 (§2.7) tested the AI layer end-to-end with a live OpenRouter/Kimi key, fixed four AI defects, and re-verified them in production (real tutor chat, quiz generation). Finally, a dedicated verification & cross-reference pass (§11) re-checked every issue against current code + live schema, separated real bugs from seed artifacts, and mapped fix blast-radius. Where a later pass changed the verdict, the document says so plainly — guesses are not presented as facts.

The report is organised so a non-author can act on it:

  • Section 1 — Executive summary & scorecard: the health of each role at a glance.
  • Section 2 — Live ground truth: the numbers Pass 2 pulled from production, and which static claims they confirmed or corrected.
  • Sections 3–7 — Per-role findings: feature-by-feature status tables + ranked issues (BLOCKER / HIGH / MEDIUM / LOW), each with evidence.
  • Section 8 — Cross-cutting infrastructure: realtime, notifications, Sentry consent, RLS, i18n, CORS, dead code.
  • Section 9 — The badge-XP saga retrospective: a candid answer to "why did this happen, why wasn't it caught, and why did the earlier approach fail?" — the questions you explicitly asked for.
  • Section 10 — Prioritised remediation plan: what to fix, in what order, with effort and risk.
  • Section 11 — Verification & cross-reference appendix: every issue re-checked against live code + live schema, real-bug vs seed-artifact, and a fix blast-radius / "don't-break-these" map.
Severity key. BLOCKER a core promised feature is non-functional in production · HIGH a feature is broken or silently wrong for real users · MEDIUM correctness/UX/security gap with a workaround · LOW hygiene, dead code, or type-safety debt.

1 · Executive summary & scorecard

The platform is broad and largely functional — auth, RLS isolation, OBE CRUD, the XP ledger, leaderboard, AI tutor, marketplace, challenges and most admin/coordinator/teacher management surfaces work end-to-end. The problems cluster in three places: (a) two PDF-generation edge functions that query a schema that no longer exists, (b) the student daily-engagement loop (streak, perfect-day, habit recording, journal/grade XP) which is wired in the UI but never actually fires the backend, and (c) a privacy-consent gap and a handful of RLS/CORS/realtime omissions in shared infrastructure.

Overall verdict: NOT yet release-clean. Confirmed BLOCKERs in accreditation output, student engagement, parent access, and privacy consent — now verified by acting as a real user of every role. None are data-loss risks and none breach multi-tenant isolation (which tested leak-free); all are fixable without schema redesign. The recently-deployed badge fix is verified working in production.
Update since first issue (fixes already deployed): the AI layer has been repaired and re-verified live — the AI tutor now holds real Kimi-powered conversations (with the academic-integrity guard working), AI quiz generation produces real questions, and the role-gate bug on ai-feedback-draft/ai-module-suggestion is fixed. See §2.7. Remaining AI gap: RAG course-material citations need an embeddings provider. The accreditation-PDF, planner, parent-RLS, engagement-XP and consent blockers below are still open.
Headline correction from live + seed-and-test: the accreditation PDF functions reject every coordinator/admin with a 403 before any query (role read from the JWT, which doesn't carry it). The same role-gate bug also breaks ai-feedback-draft. Driving the empty domains surfaced three more BLOCKERs: planner-task create/complete violates a DB CHECK (pending vs todo), AI quiz generation 404s on a non-existent courses.institution_id column, and study-session/wellness/planner/goal XP all 403 because the client awards XP as the student. The XP ledger, multiplier engine, marketplace, and multi-tenant isolation all tested healthy.

The five things to fix first

  1. Accreditation report & course-file PDFs (Admin/Coordinator BLOCKERs — TWO stacked bugs each). Live testing showed they first reject every coordinator/admin with a 403 (role read from the JWT, which doesn't carry it — it's in profiles), and even past that they query a pre-migration schema and emit empty sections. These are the artefacts an institution shows an accreditation body — currently fully unusable. generate-accreditation-report, generate-course-file
  2. Student engagement loop (Student BLOCKERs). Login doesn't advance the streak; the 4th daily habit doesn't pay the 50 XP Perfect Day; journal and grade actions advertise XP they never grant. The gamification headline numbers are seed data, not live behaviour.
  3. Sentry consent gate (Shared BLOCKER). initSentry() runs unconditionally at app load, bypassing the consent-gated path — error/replay capture starts without analytics consent.
  4. Parent data access (Parent BLOCKER). Progress & Attendance read student_courses first, which has no parent RLS policy, so the pages are empty for every parent.
  5. CORS typo + notification realtime (HIGH). x-content-type instead of x-client-info can break a browser-invoked function; the live-update bell hook is never mounted.

9 · Retrospective — the badge-XP saga (the questions you asked)

You asked, directly: "why is this causing the issue, why didn't we work to resolve this, and why did you use the bad approach earlier?" Here is the honest engineering account of the badge/XP edge-function work that immediately preceded this audit.

Q1 · What was actually broken?

Two distinct defects stacked on top of each other:

  • The original blocker (B-1): check-badges wrote earned individual badges to a student_badges table while the UI read a different badges table, and award-xp never triggered check-badges after an XP award. So badges could never appear, and the badge-check rarely ran. This is why production showed badges = 0 for 71 students with 4,119 XP transactions.
  • The hidden second defect (discovered during the fix): once badges were being awarded, the follow-on "award the badge's XP" call (and the award-xp → check-badges trigger) failed with a gateway 401. No server-to-server award-xp call had ever succeeded — visible in the ledger as zero rows for badge, tutor_engagement, challenge_reward and improvement_bonus sources.
Q2 · Why was the second defect happening? (root cause)

The functions were deployed with the platform setting verify_jwt = true, which makes Supabase's API gateway validate the Authorization: Bearer token as a JWT before the request ever reaches the function code. This project's injected SUPABASE_ANON_KEY and SUPABASE_SERVICE_ROLE_KEY are the modern non-JWT sb_… format (confirmed: the publishable key is sb_publishable_…). A non-JWT string can never pass a JWT-verification gate. So every function-to-function call — which sends one of those keys as the bearer — was rejected at the gateway with 401 before the function's own (correct) in-handler auth logic could run. The function code was fine; the platform gate in front of it was incompatible with the key format the platform itself injects.

Q3 · Why wasn't this caught earlier?
  • It was latent behind the first blocker. Because badges were never awarded end-to-end (B-1), the badge→XP sub-call was never actually exercised in production. The 401 only became reachable after B-1 was fixed. Fixing one bug exposed the next — classic layered failure.
  • The ledger evidence was never read this way before. The "zero rows for every server-to-server XP source" signal was sitting in the database the whole time, but no prior check queried the source distribution. Pass 2 of this audit is exactly that query.
  • Edge functions don't fail loudly. The calls were fire-and-forget with .catch(console.error); a 401 logged to a function-log nobody was tailing. There was no test asserting "a submission produces a badge XP row."
Q4 · Why did the earlier approach fail — twice — before the right fix?

This is the part worth being candid about. The fix went through three bearer-token attempts before the real cause was identified:

  1. Attempt 1 — send the service-role key as the bearer. Reasoning: "service role should be all-powerful." Result: gateway 401. Why it was wrong: the injected service-role key is the non-JWT sb_secret_… format, so verify_jwt rejects it.
  2. Attempt 2 — send the anon JWT as the bearer + service-role in a header. Reasoning: "the anon key is a JWT, so the gate will pass, and the handler can read the secret from a custom header." Result: still 401. Why it was wrong: on this project the anon key is also the modern non-JWT format, so it failed the same gate.
  3. Attempt 3 (correct) — deploy both functions with verify_jwt = false and rely on the already-robust in-handler auth (service-role exact-match via x-internal-auth, plus user-JWT ownership checks for student calls). Result: 200; badges and badge-XP both land; verified live with two real students.

The honest mistake: the first two attempts kept changing which token was sent, treating the symptom (a 401) as an auth-token problem. The actual variable was where auth was being enforced (gateway vs handler) and what format the platform keys were. The diagnostic that should have come first — "does the gateway return 401 even for a known-valid token, i.e. is this a gateway gate or a handler rejection?" — was run third, not first. Once that diagnostic matrix was run (legacy JWT → in-handler 403 = gateway accepts; injected key → 401 = gateway rejects), the root cause was obvious and the fix was immediate.

The general lesson (worth a steering rule). When a 401/403 appears on a server-to-server call, first determine whether it is the gateway or the handler talking, before changing any credential. A gateway 401 and a handler 403 look similar in logs but have opposite fixes. Changing tokens without that distinction is guessing. The repo's own guidance — "if an approach has failed twice, diagnose the root cause rather than making incremental patches" — is exactly the rule that was violated on attempts 1–2 and honoured on attempt 3.

Q5 · Is the fix safe? Did turning off verify_jwt weaken security?

No meaningful weakening. verify_jwt at the gateway is a coarse "is this any valid JWT" check. Both functions enforce their own, stricter auth inside the handler: award-xp and check-badges accept a service-role secret only via the x-internal-auth header (exact-match) for server-to-server calls, and for end-user calls they validate the user's JWT with getUser() and enforce ownership (a student can only act on their own id — verified live: a cross-user call returns 403). Turning off the redundant gateway gate removed the only thing blocking legitimate internal calls while leaving the real authorization intact. The change is recorded in scripts/deploy-edge-functions.sh (NO_VERIFY_JWT_FUNCTIONS) so a future batch deploy doesn't silently re-enable the gate and re-break it.

Q6 · Current state of that work

The badge fix and the verify_jwt=false redeploy are live and verified (56 individual badges, 2 badge-XP transactions — the first ever — and a confirmed no-infinite-loop). However, the edited source for check-badges, award-xp and deploy-edge-functions.sh is currently uncommitted on the local main branch and has not yet gone through a PR. Action required: move these to a feature branch and open a PR so production code matches what is deployed (and so CI/property tests run against it). This is item R-0 in the plan below.
